# TOUCH TOOLS — Complete Development Blueprint (1 of 5)
## Foundation Architecture, Header System & Navigation Redesign

---

## PREAMBLE: WHAT THIS DOCUMENT IS

This is the first of five comprehensive development directives for Touch Tools (formerly "AEHub"), a web-based After Effects remote control surface. Every specification here is derived from the source direction document, cross-referenced against the existing codebase (index.html, styles.css, app.js, server.js, bridge.jsx, package.json), and refined through critical analysis of what works, what doesn't, and what the user actually needs. The bridge.jsx, server.js, and core WebSocket-to-TCP relay architecture are **not** to be altered. Every change described herein operates within the `public/` directory (HTML, CSS, JS) unless explicitly stated otherwise.

The overarching product philosophy: Touch Tools should feel like a **professional control surface** — think of how DaVinci Resolve's hardware panels or Tangent's color grading surfaces give you tactile, immediate control over complex software. This is that, but web-hosted, touch-optimized, and purpose-built for After Effects users working via RDP, tablet, or secondary screen.

The design language borrows structural DNA from **LightMotion** (Android video editor) in terms of workflow sequencing, layer management philosophy, and progressive disclosure of controls — but adapted for AE's far more complex property model.

## 1. PROJECT NAMING & IDENTITY

### 1.1 — Canonical Name: "Touch Tools"

All references to "AEHub" throughout the codebase must be replaced. This includes:
- The `<title>` tag in index.html — currently reads "AEHub" or similar, must become "Touch Tools"
- Any `document.title` assignments in app.js
- The browser tab will display "Touch Tools"
- If there is a visible header/brand element rendered in the DOM, it must read "Touch Tools"
- Toast messages, error strings, or console logs referencing "AEHub" should be updated for consistency

### 1.2 — Visual Identity Placement

The name "Touch Tools" appears once — in the top-left of the header bar. It is **not** repeated in every panel or section. One instance, always visible, anchors the user's sense of where they are. Typography: the name should use the display font with slightly heavier weight than surrounding UI text. Not oversized. Not flashy. Professional. It sits at the same vertical baseline as the other header elements.

## 2. HEADER BAR — COMPLETE RESTRUCTURE

The current header bar has: connection status text ("AE Connected"), comp name ("comp1"), settings icon, and a "Launch AE" button. This entire layout must be restructured. The new header bar is the **primary persistent navigation strip** — it never scrolls, never hides, and contains only the most critical always-available controls.

### 2.1 — New Header Bar Layout (Left to Right)

```
[Touch Tools] [● Status Dot] [X Mode] [editable comp name + pencil] [Settings ⚙] [Undo] [Redo] [? Help]
```

Eight elements total. No text labels for status. No "Launch AE." Every element described below in exact specification.

### 2.2 — Status Dot (Connection Indicator)

**Current state:** Dot + text label ("AE Connected", "Server Only", "Disconnected"). The text is redundant — the dot's color already communicates the state.

**Change:** Remove the text label entirely. The dot alone remains. It must be a minimum of 10px diameter (accessible tap target on touch devices) with a subtle CSS `box-shadow` glow in the state color to improve visibility. The three states:

- **Green (#00e676 or similar):** AE bridge is connected. TCP socket on port 8092 is alive. Commands will execute.
- **Amber/Yellow (#ffc107):** WebSocket to server is open, but TCP bridge to AE is not responding. Commands will fail. This state means the Node server is up but bridge.jsx isn't running or isn't accepting connections.
- **Red (#ff1744):** No connection at all. Server is unreachable.

**Interaction:** No click handler on the dot itself. It is purely an indicator. No tooltip needed — the color semantics are universally understood. However, if the connection is in red or amber state for more than 5 seconds, a subtle CSS pulse animation (keyframe opacity oscillation at 1s interval) should activate on the dot to draw attention to the disconnected state. This animation stops once green is restored.

**Technical note:** The existing `updateStatus()` function in app.js currently sets a `statusText` element's innerHTML. Refactor this to only toggle CSS classes on the dot element (`.dot-connected`, `.dot-server`, `.dot-disconnected`). Remove any DOM reference to the status text element.

### 2.3 — X Mode Button

**Position:** Immediately to the right of the status dot.

**Visual:** A compact rectangular button, same height as the header bar, with "X" as label. Slightly different surface color than surrounding elements — use a muted variant of the accent color or a subtle border to differentiate it. When inactive, it blends with the header. When active (X Mode panel is open), it gets a highlighted border-bottom or background change to indicate the active state.

**Behavior:** Clicking X Mode activates the X Mode panel in the content area (described in detail in Message 4). It is **not** a tab in the bottom tab strip — it is a header-level action because it's an experimental feature that overlays or replaces the current view contextually.

**State persistence:** X Mode's active/inactive state should be stored in a variable (not localStorage — it should reset on page reload since it's experimental and we don't want users accidentally stuck in an experimental mode).

### 2.4 — Editable Composition Name

**Current state:** The comp name ("comp1") is displayed as static text that updates when you select a different comp from the dropdown.

**Change:** The comp name becomes an **inline-editable field**. Display it as text normally. On click (or tap), it transforms into an inline `<input>` with the current name pre-selected. The user can type a new name. On **Enter** keypress or **blur** (clicking elsewhere), the new name is sent to AE via a `renameComp` command. On **Escape** keypress, the edit is cancelled and the original name is restored.

**Visual treatment:** A small pencil/edit icon (Unicode ✏ or SVG) appears to the right of the name on hover (desktop) or always visible (touch devices, since there's no hover). Clicking the pencil also triggers edit mode. The editable field should have the same font and size as the display text to avoid layout shift when transitioning between display and edit modes. Use `min-width` on the input to prevent it from shrinking to zero when empty.

**Bridge dependency:** This requires a `renameComp` command in bridge.jsx. The handler would be: `comp.name = args.newName; return {success: true, name: comp.name};`. **Since we cannot modify bridge.jsx**, this feature should be coded into app.js with the command already structured correctly, and a TODO comment marking where the bridge handler needs to be added. If the bridge doesn't have this handler, the command will timeout gracefully (existing timeout mechanism handles this). When the user later adds the bridge handler, the feature lights up automatically.

**Sync behavior:** When the active comp changes (via dropdown selection), the display name updates immediately. If the user is mid-edit and a comp change happens programmatically, cancel the edit and show the new comp name.

### 2.5 — Settings Gear Icon

**Current state:** Already exists, opens comp settings panel.

**Change:** No functional change. Retain the gear icon. Ensure it has a toggle behavior — clicking when settings are open closes them, clicking when closed opens them. The icon should have a subtle rotation animation on click (15-20 degree rotation, 200ms ease) as visual feedback.

**Refinement:** The settings panel itself undergoes significant changes (covered in Message 2), but the trigger mechanism stays the same.

### 2.6 — Undo Button

**Position:** To the right of the settings gear. Small, compact, square button with a standard undo arrow icon (↩ or ↶ Unicode, or an inline SVG path).

**Current state:** Does not exist in the UI. The bridge has `beginUndoGroup`/`endUndoGroup` but no dedicated undo command.

**Implementation:** Wire the button to send `executeCommand` with AE's internal command ID for Undo, which is `16`. This uses the existing `executeCommand` handler in bridge.jsx — no bridge modification needed. The command structure: `send({command: 'executeCommand', args: {commandId: 16}})`.

**UX refinement:** After executing undo, trigger a property refresh on the currently selected layer (if any) so the UI reflects the undone state. Chain this onto the command promise: `.then(() => loadLayerProperties())`.

**State feedback:** Optionally, disable the button visually if the last operation was not undoable (bridge returns an error or the undo stack is empty). However, AE doesn't expose its undo stack depth via scripting, so the button should always appear enabled. If the undo fails silently (AE does nothing), that's acceptable — the user just clicks and nothing happens, which is standard behavior.

### 2.7 — Redo Button

**Position:** Immediately to the right of the Undo button. Same size, same visual treatment, with a redo arrow icon (↪ or ↷).

**Implementation:** Same as undo, but using AE's Redo command ID `17`. Command structure: `send({command: 'executeCommand', args: {commandId: 17}})`.

**Same post-execution behavior:** refresh properties if a layer is selected.

### 2.8 — Help/Question Mark Button

**Position:** Far right of the header bar.

**Visual:** A circled question mark (ⓘ or ? in a small circle). Subdued color — muted text color, not accent color. It should not draw attention away from functional controls.

**Behavior:** Clicking opens an **interactive guided tour overlay** (described below). This is a low-priority feature — implement the button placeholder now, wire the overlay content later.

### 2.9 — Header Bar CSS Specification

- `position: sticky; top: 0; z-index: 100;` — always visible, above all content
- Height: 40-44px on desktop, 48px on touch (larger tap targets)
- `display: flex; align-items: center; justify-content: space-between;` with appropriate gap
- Background: slightly elevated surface color (1-2 brightness steps above body background)
- `border-bottom: 1px solid` with a subtle separator color
- `padding: 0 12px;`
- All interactive elements within should be `flex-shrink: 0` to prevent compression

## 3. NAVIGATION ARCHITECTURE — TAB SYSTEM RESTRUCTURE

### 3.1 — Current Tab Strip

Currently: Layers, Transform, Effects, Shapes, Camera, Text, Masks, Keyframes, Console.

### 3.2 — Revised Tab Strip

**New order:** `Layers | Transform | Effects | Shapes | Camera | Text | Masks | Keyframes | X Mode`

Changes:
- **"Console" is renamed to "X Mode"** — this is where the experimental extension/plugin/script integration features live
- The tab strip must be **horizontally scrollable** on small viewports. Currently, if the viewport is narrow, tabs may overflow and become inaccessible. Fix: wrap the tab strip in a container with `overflow-x: auto; overflow-y: hidden; white-space: nowrap;` and hide the scrollbar with CSS (`-webkit-scrollbar: display: none;` for clean appearance while maintaining scroll functionality).
- On touch devices, horizontal swipe on the tab strip should scroll through tabs. Native overflow scrolling handles this if the CSS is correct.
- Each tab button should have `flex-shrink: 0` so they maintain their full width and don't compress.
- The active tab indicator (bottom border or background change) must animate smoothly between tabs — use `transition: background 0.2s ease, border-color 0.2s ease`.

### 3.3 — Tab Content Panels

Each tab reveals a content panel below the tab strip. The content panel occupies the remaining vertical space between the tab strip and the timeline/transport area (described in Message 3). Key behavioral requirements:

- **Only one panel visible at a time.** When a tab is clicked, the previous panel hides (`display: none`) and the new panel shows. No transition animation needed for panel swaps — instant toggle. Animations on content panels create perceived latency.
- **Lazy loading:** Panel data should only be fetched when the tab becomes active. Don't pre-load effects list when the user hasn't clicked the Effects tab. The existing `onTabSwitch` callback mechanism handles this — ensure each tab's switch handler triggers its respective `load*()` function.
- **Scroll containment:** Each panel must have its own `overflow-y: auto` with `max-height` calculated from the available viewport space. Do NOT let the body scroll — only the active panel scrolls. This is critical for the RDP/touch use case where you don't want the entire page moving when you're trying to scroll through effects.

### 3.4 — Layers Panel — Special Treatment

The Layers panel is the **default active tab** on page load. It is also the most frequently used panel. It gets special treatment described in detail in Message 2, but the key architectural point is: the Layers panel contains both the layer list AND inline transform controls. It is not a flat list that requires you to switch to a separate "Transform" tab to adjust a layer's properties. The Transform tab becomes a **supplementary view** for advanced/focused transform work, while basic transform controls (position, rotation, scale, opacity) are accessible directly within the Layers panel when a layer is expanded.

### 4.1 — Composition Settings Panel (Collapsible)

**Trigger:** The gear icon in the header bar toggles this panel's visibility.

**Behavior:** When closed (default), the panel collapses to `height: 0; overflow: hidden;` with a CSS transition on height (200ms ease). When open, it expands to its natural content height. The content below (comp selector, timeline, tabs, panel) shifts down smoothly — use `transition: margin-top 0.2s ease` on the sibling container.

**Important:** This panel is NOT inside any tab. It is a **persistent overlay** between the header and the content area. It affects composition-level settings that are relevant regardless of which tab is active.

Full specification of what goes inside this panel is in Message 2.

### 4.2 — Composition Selector + Layer Toolbar

This is the row containing the comp dropdown and the layer creation buttons. Current layout has "Layers" text label, dropdown, refresh, and creation buttons. Revised layout:

**Remove:** The "Layers" text label. The tab strip already identifies this section. The text is redundant and consumes horizontal space.

**Comp dropdown:** Stays on the left. Full width available minus the button row.

**Refresh button:** Remove text label. Icon only — a circular arrow (↻) inside a small square button. This frees horizontal space for the creation buttons.

**Layer creation buttons:** In order: `Solid | Shape▾ | Text | Null | Camera | Light`

- **Solid:** Direct action. Click → creates solid layer → auto-selects it. No dialog (the default solid settings are fine; user can adjust properties after).
- **Shape:** This button has a **dropdown** (not a direct action). Clicking reveals a menu of preset shape types: Rectangle, Ellipse, Polygon, Star, Line. Each option creates a shape layer with the corresponding path operator pre-added. The dropdown should use a `position: absolute` overlay with proper z-index, and dismiss on click-outside or Escape. This is similar to LightMotion's shape picker where you get multiple options from a single button.
- **Text:** Direct action. Click → creates text layer → auto-selects it.
- **Null:** Direct action. Click → creates null object → auto-selects it.
- **Camera:** Direct action. Click → creates camera → auto-selects it.
- **Light:** Direct action. Click → creates light → auto-selects it.

**Post-creation behavior (ALL buttons):** After creating any layer, the layer list must refresh (`loadLayers()`), the new layer must be auto-selected (`selectLayer(newIndex)`), and the transform/property panel must load for the new layer (`loadLayerProperties()`). This is the "immediate edit" workflow — the user shouldn't have to manually find and click their newly created layer.

### 4.3 — Timeline Bar

**Position:** Below the comp selector toolbar, above the tab strip. Always visible (not inside any tab). This is a critical UX improvement — the timeline provides temporal context at all times.

**Layout:**

```
[⏮] [◀] [▶/⏸] [▶] [⏭]  |  00:00:15:12  |  [====playhead scrubber====]  |  45/120
```

**Transport buttons:** Go to start, previous frame, play/pause (toggle), next frame, go to end. These are the SAME buttons currently in the footer transport bar — they move to this location. The footer transport bar is **removed entirely**.

**Play/Pause fix:** The play/pause button must:
1. Send the `playPause` command to AE
2. Track playing state locally (boolean `isPlaying`)
3. Toggle its icon between play (▶) and pause (⏸) based on that state
4. Poll `getCurrentTime` every 500ms when playing (existing behavior) — but ALSO poll to detect if AE stopped playing externally (user clicked in AE itself). If `getCurrentTime` returns the same value two polls in a row while `isPlaying` is true, assume AE stopped and toggle the icon back to play.

**Timecode display:** HH:MM:SS:FF format, monospace font, reads from `getCurrentTime` response. Updates every 500ms during playback. Updates immediately after frame-step buttons are pressed.

**Playhead scrubber:** A horizontal track representing the full duration of the composition. The current time is indicated by a draggable handle (small vertical line or diamond). The track should show the total duration as its width. Dragging the handle sends `setCurrentTime` to AE. While dragging, timecode display updates in real-time. On release, final time is committed. The scrubber must be touch-friendly: minimum 44px hit area height, even if the visual track is thinner.

**Frame counter:** "X/Y" format showing current frame / total frames.

**Technical implementation for scrubber:**
- Calculate handle position: `(currentTime / totalDuration) * trackWidth`
- On pointerdown on the track or handle: capture pointer, begin drag
- On pointermove: calculate `(pointerX - trackRect.left) / trackRect.width * totalDuration`, clamp to [0, totalDuration], send `setCurrentTime` (throttled to 60fps via RAF)
- On pointerup: release capture, final time already committed
- The track background could optionally show a thin line with keyframe markers as small dots — but this is a nice-to-have, not essential for the initial implementation

## 5. CONNECTION SYSTEM — REFINEMENTS

### 5.1 — Auto-Reconnect Behavior

The existing auto-reconnect with 2500ms retry is fine. But the connection flow needs refinement:

**On page load:** Do NOT auto-connect. Wait for the user to click Connect (or press Enter in the IP field). This prevents error toasts on every page load when the server isn't running.

**On disconnect:** Show a toast: "Connection lost. Reconnecting..." The status dot turns red. Auto-reconnect attempts continue in the background. Each failed attempt is silent (no toast). On successful reconnect, show: "Reconnected" with green toast. The dot returns to green.

**On manual connect:** If the IP/port fields are empty, use defaults (localhost:8092). If they have values, use those.

### 5.2 — Remove "Launch AE" Button

**Current state:** There is a "Launch AE" button in the header or connection area.

**Change:** Remove it entirely from the HTML. Remove its event listener from app.js. Remove any associated CSS. The logic is sound — a user of this tool will have already launched AE before opening the browser interface. The button adds clutter and confuses the mental model (does it launch AE? Does it connect? These are different operations).

### 5.3 — Remove "AE Connected" Text

As described in section 2.2, remove the text label from the connection status indicator. The dot alone communicates state. Any remaining references to status text in the DOM (span elements, innerHTML assignments) should be removed or repurposed.

### 5.4 — Connection Info Display

The IP input and port input should be styled compactly. On mobile/touch, these should be large enough to tap (44px minimum height). The Connect button should show "Connect" when disconnected and "Disconnect" when connected — toggling label based on state.

## 7. JAVASCRIPT ARCHITECTURE — STATE MANAGEMENT

### 7.1 — Central State Object

The current codebase uses scattered global variables. This creates synchronization bugs — when the selected layer changes, multiple panels need to update, but there's no central mechanism to coordinate this.

**Introduce a `state` object:**

```javascript
const state = {
  connection: 'disconnected',  // 'disconnected' | 'server' | 'ae'
  currentCompId: null,
  currentCompName: '',
  selectedLayerIndex: -1,
  activeTab: 'layers',
  isPlaying: false,
  currentTime: 0,
  totalDuration: 0,
  totalFrames: 0,
  frameRate: 30,
  xModeActive: false,
  settingsOpen: false
};
```

### 7.2 — State Change Notification System

Implement a simple pub/sub mechanism. When any state value changes, registered listeners are notified:

```
function setState(key, value) {
  const oldValue = state[key];
  state[key] = value;
  if (listeners[key]) {
    listeners[key].forEach(fn => fn(value, oldValue));
  }
}

function onStateChange(key, callback) {
  if (!listeners[key]) listeners[key] = [];
  listeners[key].push(callback);
}
```

**Usage example:** When `selectedLayerIndex` changes:
- `loadLayerProperties()` is called (transform panel updates)
- If effects tab is active, `loadEffects()` is called
- If shapes tab is active, `loadShapes()` is called
- Layer list highlights the correct row
- Trackpad resets its cached values

This eliminates the current pattern of manually calling multiple functions after every layer selection change, which is error-prone and easy to forget when adding new features.

### 7.3 — Command Wrapper

Standardize all AE commands through a single wrapper that handles loading state, errors, and follow-up actions:

```
async function aeCommand(command, args = {}, options = {}) {
  const { showLoading = false, toastOnError = true, onSuccess = null } = options;
  
  if (showLoading) showSpinner();
  
  try {
    const response = await send({ command, args });
    if (onSuccess) onSuccess(response);
    return response;
  } catch (err) {
    if (toastOnError) showToast(`Error: ${err.message}`, 3000);
    throw err;
  } finally {
    if (showLoading) hideSpinner();
  }
}
```

This replaces the current pattern where each function manually calls `send()`, manually handles `.then()` and `.catch()`, and manually shows/hides loading indicators. Every command flows through one path, making error handling, logging, and debugging centralized.

## 8. EVENT DELEGATION — PERFORMANCE PATTERN

### 8.1 — Current Problem

The current codebase likely attaches individual event listeners to each layer row, each property control, each button. For a layer list with 50 layers, that's 50+ event listeners just for click handlers. For a property tree with 100 properties, that's 100+ listeners for scrub dials, 100+ for keyframe buttons, etc. This has memory implications and creates maintenance headaches.

### 8.2 — Solution: Event Delegation

Attach a single event listener to each **container** (layer list container, property panel container, tab strip, etc.) and use event target inspection to determine which element was interacted with:

- Layer list container: one click listener. On click, check `e.target.closest('[data-layer-index]')` to get the layer index. Check `e.target.closest('[data-action="split"]')` for action buttons.
- Property panel container: one pointerdown listener for all scrub dials. Check `e.target.closest('.scrub-dial')` to identify the specific dial.
- Tab strip: one click listener. Check `e.target.closest('[data-tab]')` to identify which tab.

This reduces hundreds of event listeners to a handful. It also automatically handles dynamically added elements — new layers added to the list don't need new event listeners because the container listener catches everything.

## 9. RENAME COMPOSITION — BRIDGE WORKAROUND

### 9.1 — The Problem

The `renameComp` command doesn't exist in bridge.jsx and we cannot modify bridge.jsx.

### 9.2 — The Workaround

Use the existing `executeScript` command to run raw ExtendScript. The rename can be achieved by sending:

```
send({command: 'executeScript', args: {script: 'app.project.activeItem.name = "NewName"'}})
```

This is a valid workaround that uses existing infrastructure. After executing, update the local state and UI display to reflect the new name. If the script fails (e.g., active item isn't a comp), the error comes back through the normal error handling path.

### 9.3 — Edge Case

If the user renames a comp that is also selected in the dropdown, the dropdown option text needs to update. After the rename command resolves, call `loadComps()` to refresh the dropdown, or more efficiently, just update the specific `<option>` text in the dropdown by matching the comp ID.

## 10. GUIDED TOUR SYSTEM (LOW PRIORITY)

### 10.1 — Mechanism

When the user clicks the "?" button in the header, an overlay activates. A semi-transparent backdrop covers the entire viewport (`background: rgba(0,0,0,0.6); position: fixed; inset: 0; z-index: 999`). A highlight "window" is cut out over the target element using `box-shadow` on a positioned element or using CSS `clip-path`. Around the highlight, instructional text appears explaining what that area does.

### 10.2 — Tour Steps

Define an ordered array of tour steps, each with:
- `target`: CSS selector for the element to highlight
- `title`: Short heading (e.g., "Layer Panel")
- `content`: Explanation text (2-3 sentences max)
- `position`: Where to place the tooltip relative to the target (top, bottom, left, right)

Steps would be:
1. Header bar (connection, undo/redo)
2. Comp settings (how to configure composition)
3. Comp selector (how to switch comps)
4. Timeline (how to navigate time)
5. Layer creation buttons (how to add layers)
6. Layer list (how to select and manage layers)
7. Tab strip (what each tab contains)
8. Transport controls (play/pause, frame stepping)

### 10.3 — Navigation

"Next" and "Previous" buttons in the tooltip. "Skip" or "×" button to dismiss. Progress dots showing which step of N you're on. On completion, dismiss and don't show again (store in localStorage).

### 10.4 — Priority

This is explicitly **low priority**. Implement the "?" button with a placeholder that shows a simple modal saying "Guided tour coming soon" for now. Build the actual tour system in a later iteration.

## 11. RESPONSIVE BEHAVIOR

### 11.1 — Desktop (1920px+)

Full layout as described. All panels visible. Scrub dials and trackpad at comfortable sizes.

### 11.2 — Laptop/Tablet (1024-1919px)

### 11.3 — Small Tablet/RDP Window (768-1023px)

### 11.4 — Phone/Small (below 768px)

**Implementation:** Use CSS media queries at these breakpoints. Do NOT use JavaScript for layout changes — CSS handles responsive behavior more performantly and doesn't require re-renders.

## 12. ANIMATION & TRANSITION POLICY

### 12.1 — Permitted Animations

- Panel expand/collapse (comp settings): `height` transition, 200ms ease
- Tab active indicator: `transform` or `background-color`, 150ms ease
- Status dot pulse (disconnected state): opacity keyframe, 1s infinite
- Button hover states: `background-color`, 150ms ease
- Toast appearance: `transform: translateY` + `opacity`, 200ms ease in, 300ms ease out
- Settings gear rotation on click: `transform: rotate(20deg)`, 200ms ease
- Dropdown menu appearance: `opacity` + `transform: scaleY`, 150ms ease

### 12.2 — Forbidden Animations

- No page load animations (fade-up, slide-in). They create perceived latency.
- No animation on scrub dial value changes — instant visual update.
- No animation on layer list refresh — instant DOM replacement.
- No skeleton loaders — use a simple text "Loading..." or a small spinner.
- No parallax, no scroll-linked animations, no decorative motion.

### 12.3 — Performance Rule

Any animation that triggers layout recalculation (width, height, top, left, margin, padding) must be reviewed for performance. Prefer `transform` and `opacity` animations which are GPU-composited. If an animation causes jank on a low-end RDP session, remove it. Performance in remote desktop scenarios is the priority — not visual flair.

## 13. COMPOSITION SETTINGS PANEL — FULL SPECIFICATION

### 13.1 — Panel Trigger & Behavior

The comp settings panel is toggled by the gear icon in the header bar. It is a **collapsible panel** that sits between the header bar and the comp selector row in the vertical layout sequence. When closed, it has zero height and is completely hidden. When open, it expands to its natural content height with a CSS transition. The content below shifts down smoothly — no overlap, no floating overlay, no z-index wars.

**Collapse mechanism:** Use `max-height: 0; overflow: hidden; transition: max-height 0.25s ease;` for the closed state. On open, set `max-height` to a value larger than the panel's natural height (e.g., `600px`). This is a well-established CSS pattern that avoids animating `height: auto` (which isn't animatable). The content below should have a corresponding `transition: margin-top 0.25s ease` or simply be wrapped in a container that uses flexbox with `flex-shrink: 0` on the settings panel.

**State persistence:** The open/closed state should NOT persist across page reloads. Default is always closed. The user opens it when they need it, and it closes when they're done. This keeps the default view clean.

### 13.2 — Internal Layout

The settings panel uses a **two-column grid** layout internally. Left column: composition-level parameters. Right column: toggle switches and quick actions. The grid should be `display: grid; grid-template-columns: 1fr 1fr; gap: var(--space-lg);` with responsive collapse to single-column below 768px viewport width.

### 13.3 — Left Column: Composition Parameters

**Resolution Group:**
- Two inputs side by side: Width and Height
- Labeled "W" and "H" with small label text above or inside the input as placeholder
- Input type: `number`, `min: 1`, `max: 30000`, `step: 1`
- Below the inputs: a row of **preset buttons** — compact, horizontally arranged
  - `16:9` (maps to 1920×1080)
  - `9:16` (maps to 1080×1920)
  - `1:1` (maps to 1080×1080)
  - `2K` (maps to 2560×1440)
  - `4K` (maps to 3840×2160)
  - `Custom` (does nothing — just a label indicating current values don't match any preset)
- When a preset is clicked, both width and height inputs update. The values are NOT sent to AE immediately — they wait for the "Apply" action (see below).
- The currently active preset (if resolution matches) should have an `active` visual state (accent border or background).

**Frame Rate:**
- Input type: `number`, `min: 1`, `max: 120`, `step: 0.001` (to support 23.976, 29.97, 59.94)
- Common presets as clickable chips below: `23.976 | 24 | 25 | 29.97 | 30 | 50 | 59.94 | 60`
- Same deferred-apply behavior

**Duration:**
- Input type: `number`, `min: 0.01`, `max: 10800`, `step: 0.01`
- Display in seconds. Optionally, show the equivalent timecode next to the input (computed client-side from duration and frame rate) as a read-only label.
- Same deferred-apply behavior

**Background Color:**
- A color swatch (small square, 24×24px) that shows the current background color
- Clicking the swatch opens a native `<input type="color">` picker (positioned off-screen or overlaid)
- Below: a row of **preset color chips** — small 20×20 squares
  - Black, White, Red, Green, Blue, Yellow, Cyan, Magenta, Gray (mid-gray)
  - Plus a "Transparent" chip — visually represented as a checkerboard pattern (standard transparency indicator)
- Clicking a preset chip updates the swatch preview immediately. Does NOT send to AE until Apply.
- When "Transparent" is selected, the swatch shows the checkerboard, and the `executeCommand(2573)` will be sent on Apply (this toggles AE's "preserve underlying transparency" or background transparency — the exact command may vary, but the existing codebase uses 2573).

### 13.4 — Right Column: Toggles & Quick Actions

**Toggle Switches:**
Each toggle is a labeled switch row. Use a custom CSS toggle (a small track with a sliding dot, 36×20px). The label sits to the left, the switch to the right.

- **Motion Blur** — toggles composition-level motion blur. This is already implemented in the codebase. Wire to the existing `setMotionBlur` command.
- **Shy Layers** — toggles the "Hide Shy Layers" composition switch. Maps to `comp.hideShyLayers`. Command: `setCompSettings` with a `shyLayers` boolean argument.
- **Collapse** — toggles the collapse transformation switch for the composition. Less commonly used but belongs here.
- **Frame Blending** — toggles frame blending at the comp level.

Each toggle should read its current state when the settings panel opens (fetch from `getCompSettings` response) and reflect it visually (switch position). Toggling sends the command immediately — these are instant-apply, unlike the deferred parameters above.

**Preview Zoom:**
- A dropdown or segmented control for preview resolution: `Full | Half | Third | Quarter`
- Maps to AE's `comp.resolution`. Command: `setCompSettings` with a `resolution` argument (values: 1, 2, 3, 4).
- This is also instant-apply on selection change.

### 13.5 — Apply Button

**Critical design decision:** The resolution, frame rate, duration, and background color inputs are **batched** and not sent individually. At the bottom of the settings panel, spanning the full width, there is a single "Apply Settings" button.

When clicked:
1. Validate all inputs (positive numbers, within range)
2. Send a single `setCompSettings` command with all parameters bundled: `{width, height, duration, frameRate, bgColor}`
3. If the transparent background toggle is active, send `executeCommand(2573)` as a follow-up
4. Show a toast: "Composition settings updated"
5. Close the panel (collapse it)

**Why batch instead of live-apply:** Changing resolution in AE triggers a canvas resize which is computationally expensive. Firing this on every keystroke would be destructive. The batch approach with an explicit Apply button gives the user confidence that they're making a deliberate change.

### 13.6 — Reset Button

Next to (or below) the Apply button, a "Reset" button that reverts all inputs to the current comp settings (re-reads from AE). This is useful if the user has made changes they don't want to apply — click Reset and the fields snap back to the live values.

## 14. LAYER SYSTEM — COMPLETE REDESIGN

The layer system is the **most interacted-with component** of Touch Tools. It must be fast, clear, and information-dense without being cluttered. The current implementation has a basic layer list with action buttons. The redesigned system is significantly more capable.

### 14.1 — Layer Row Structure

Each layer in the list is a **row** with the following elements, left to right:

```
[index] [type icon] [visibility toggle] [name (inline-editable)] [expand/collapse chevron]
```

Below the row (when expanded):
```
[mini transform controls: position XY, rotation, scale %, opacity %]
[action buttons: split | duplicate | delete | solo | lock | shy]
```

### 14.2 — Layer Index

A small numeric label on the far left. Shows the layer's index in the AE composition (1-based, matching AE's panel). This is a read-only display — not editable. Font: monospace, small size (10-11px), muted color. The index helps users correlate layers between Touch Tools and AE.

### 14.3 — Type Icon

A small icon (16×16px or so) indicating the layer type. Use Unicode or simple inline SVG for each type:

- **AV (footage/video):** A small film frame icon or ▶ symbol
- **Solid:** A filled square
- **Text:** "T" or an Aa symbol
- **Shape:** A small pentagon or circle-outline
- **Null:** A small crosshair or target
- **Camera:** A small camera icon or ◎
- **Light:** A small sun or bulb icon or 💡 outline
- **Adjustment layer:** A small slider icon or ✦
- **Precomp:** A small nested rectangle icon

The icon provides instant visual recognition of layer type without reading text. Color-code subtly: footage layers in default text color, shape/solid in a slightly tinted variant, cameras and lights in accent-muted color.

### 14.4 — Visibility Toggle

An **eye icon** (👁 or 👁‍🗨 outline) that toggles the layer's visibility in AE. This maps to `layer.enabled`. Clicking toggles the state: visible = eye icon in normal color, hidden = eye icon in muted/strikethrough style. Command: `setLayerProperty` with path `ADBE Root Vectors Group` — actually, more precisely, it should send a dedicated `toggleVisibility` command or use the existing `setLayerProperty` targeting the `enabled` property of the layer.

**Visual feedback:** When a layer is hidden, the entire layer row should dim (reduce opacity to 0.4-0.5) to give immediate visual feedback that the layer is disabled. This is a CSS class toggle, not an opacity property set via JS.

### 14.5 — Layer Name (Inline-Editable)

The layer name occupies the remaining horizontal space in the row (flex-grow: 1). It displays as normal text. On click (single click that does NOT select the layer — see selection behavior below), the name transforms into an inline input field, pre-populated with the current name, text selected. On Enter or blur: send `setLayerProperty` targeting the layer's `ADBE Name` or use `layer.name = newName` via a dedicated `renameLayer` command. On Escape: cancel edit, restore original name.

**Important interaction conflict resolution:** A single click on the layer row selects the layer AND starts name edit. These must be separated:
- **Single click on the name TEXT specifically:** enters edit mode (name becomes input)
- **Single click on the row OUTSIDE the name (index, icon, toggle, empty space):** selects the layer
- **Double-click on name:** also enters edit mode (redundant but expected by users)

To implement this cleanly: the name span has its own click handler with `e.stopPropagation()` to prevent the row's selection handler from firing. The row's click handler selects the layer. This way, clicking the name doesn't trigger selection (it's already the selected layer if you're clicking its name, or it selects and immediately enters edit mode — but the former is the expected case).

**Alternative approach (simpler):** Click anywhere on the row selects the layer AND the name becomes editable only on a **second click** (when the layer is already selected). This is a common pattern in file managers (Windows Explorer, macOS Finder): first click selects, second click on the name enters rename mode. This avoids the need for `stopPropagation` and separate handlers.

I recommend the **second-click approach** for its simplicity and alignment with OS conventions. Implementation:
1. On row click: `selectLayer(index)`. Store `selectedLayerIndex` in state.
2. If `selectedLayerIndex` was already this layer's index (i.e., clicking an already-selected layer), AND the click target is within the name span, enter edit mode.
3. On blur/Enter/Escape: exit edit mode, send rename if applicable.

### 14.6 — Expand/Collapse Chevron

A small chevron icon (▸ when collapsed, ▾ when expanded) on the right side of the row. Clicking it toggles the expanded state of that specific layer. **Only one layer can be expanded at a time** — expanding one collapses all others. This prevents the layer list from becoming extremely long when multiple layers are expanded.

**Expanded content** is rendered in a sub-container directly below the layer row, indented slightly (16px left padding) to visually subordinate it to the parent row.

### 14.7 — Expanded Layer: Mini Transform Controls

When a layer row is expanded, it reveals **inline mini-transform controls**. These are compact versions of the full transform panel, showing only the most commonly needed properties:

**Position (X, Y):**
- Two small number inputs side by side, labeled "X" and "Y"
- Pre-populated with the layer's current position values
- On change (blur or Enter): send `setPropertyValue` for position
- These are compact inputs (width: ~60px each), not full scrub dials — the scrub dials are in the dedicated Transform tab

**Rotation (degrees):**
- Single number input, labeled "R"
- On change: send `setPropertyValue` for rotation
- Input width: ~50px

**Scale (%):**
- Single number input showing uniform scale as a percentage (if X and Y scale are equal). If they differ, show "X%, Y%" or just show the X scale with a note that it's non-uniform.
- On change: send `setPropertyValue` for scale (convert percentage back to 0-100 AE scale values, or if your bridge already handles this, send the raw percentage)

**Opacity (%):**
- Single number input, labeled with a small opacity icon or "Op"
- Range: 0-100
- On change: send `setPropertyValue` for opacity

**Why inline controls matter:** The most common operation after selecting a layer is tweaking its position or opacity. Forcing the user to switch to the Transform tab for every minor adjustment is workflow friction. The inline controls handle 80% of cases; the full Transform tab handles the remaining 20% (anchor point, per-component scale, advanced properties, expression management).

### 14.8 — Expanded Layer: Action Buttons

Below the mini transform controls (or to the right of them, depending on available width), a row of **action buttons**:

- **Solo** — toggles solo flag. Maps to `layer.solo`. Button has an active state (accent background) when solo is on.
- **Lock** — toggles lock flag. Maps to `layer.locked`. Active state when locked. When a layer is locked, the entire row should show a lock icon overlay and resist interaction (prevent property edits, but allow unlock toggle).
- **Shy** — toggles shy flag. Maps to `layer.shy`. Active state when shy is on.
- **Split** — splits layer at current playhead time. Command: `splitLayer`. After split, refresh layer list.
- **Duplicate** — duplicates the layer. Command: `duplicateLayer`. After duplicate, refresh and select the new layer.
- **Delete** — deletes the layer. Command: `deleteLayer`. Before executing, optionally show a confirmation (simple: "Delete [layer name]?" with Yes/No). After delete, refresh layer list and select the next available layer (or the one above).

**Button sizing:** These should be compact — icon-only on small viewports, icon+label on large viewports. Use CSS media queries or `min-width` conditions to decide.

### 14.9 — Layer Selection Behavior

**Single click on a row** (anywhere except the name, if the layer is already selected):
1. Deselect all other layers in AE: send `selectLayer` command which does `nullLayer.selected = false` for all layers, then `targetLayer.selected = true`
2. Update `state.selectedLayerIndex`
3. The state change notification system triggers:
   - Transform panel reloads if visible
   - Effects panel reloads if its tab is active
   - Shapes panel reloads if its tab is active
   - Camera panel reloads if its tab is active
   - Text panel reloads if its tab is active
   - Masks panel reloads if its tab is active
   - Trackpad resets its cached values
   - Layer row gets the `selected` CSS class

**Selected visual state:** The selected layer row has a distinct background color (slightly lighter than the surface, or a subtle accent tint) and a left border accent (2px solid, accent color). No other row has this styling.

**Keyboard navigation (nice-to-have):** When the layer list has focus, Arrow Up/Down should select the previous/next layer. Enter on a selected layer should expand it. This improves keyboard accessibility for desktop users.

### 14.10 — Layer List Virtualization Consideration

For compositions with many layers (50+), DOM rendering can become slow. If you notice performance issues, implement a simple virtualization: only render layer rows that are within the visible scroll viewport plus a buffer of ~10 rows above and below. As the user scrolls, rows outside the buffer are removed and new ones are inserted.

However, this is a **performance optimization for later**. Don't implement it preemptively. If your typical use case is under 50 layers (which is common for most AE projects), the simple approach of rendering all rows works fine. If users report lag with 100+ layer comps, add virtualization then.

### 14.11 — Layer Type Icons — Implementation Note

The bridge's `getLayers` command returns layer type information. Based on the existing codebase, the layer data likely includes a `type` field or the layer's match name (e.g., "ADBE Camera Layer", "ADBE Light Layer", "ADBE Text Layer", "ADBE Vector Layer", "ADBE AV Layer", "ADBE Null Layer").

Create a mapping function in app.js:

```
function getLayerTypeIcon(typeOrMatchName) {
  // Map AE layer types to icon characters/SVGs
  // Return the appropriate icon markup
}
```

This function is called during `loadLayers()` when rendering each layer row. It takes the type identifier from the bridge response and returns the icon markup.

## 16. CONTEXT MENU SYSTEM

### 16.1 — Purpose

Right-clicking (or long-pressing on touch) a layer row should open a context menu with layer operations. This mirrors the behavior of AE's own right-click context menu on layers, but provides only the operations Touch Tools supports.

### 16.2 — Menu Items

The context menu appears at the cursor position (or centered above the long-pressed element on touch). It contains:

**Layer Operations:**
- Select Layer
- Rename Layer
- Duplicate Layer
- Delete Layer
- Split at Playhead

**Layer Settings:**
- Toggle Solo
- Toggle Lock
- Toggle Shy
- Toggle Visibility
- Toggle Motion Blur (layer-level)
- Toggle 3D Layer
- Toggle Adjustment Layer

**Advanced:**
- Go to Layer In Point (sends `setCurrentTime` with layer's in point time)
- Go to Layer Out Point
- Set as Trackpad Target (selects this layer for trackpad operations without necessarily making it the "selected" layer in AE — useful for quick adjustments)

### 16.3 — Implementation

**Desktop:** Attach a `contextmenu` event listener to the layer list container. On trigger, prevent the default browser context menu, determine which layer row the event occurred on (using `e.target.closest('[data-layer-index]')`), position a custom context menu element at `e.clientX, e.clientY`, and populate it with the items above.

**Touch:** Use a `touchstart` + `setTimeout` pattern (long-press detection, 500ms hold). On long-press, show the context menu centered above the touch point. Cancel the long-press timer on `touchmove` (finger moved = scrolling, not long-pressing) or `touchend`.

**Dismiss:** Click/tap anywhere outside the menu dismisses it. Escape key dismisses it. After selecting a menu item, the menu closes and the action executes.

**Positioning edge cases:** If the context menu would overflow the viewport (e.g., right-clicking near the bottom edge), reposition it above the cursor instead of below. If near the right edge, shift it left. Simple clamping logic: `menuX = Math.min(e.clientX, window.innerWidth - menuWidth)` and same for Y.

## 19. LAYER REORDERING

### 19.1 — Current State

The codebase has a `reorderLayer` command in the bridge, but there may not be UI for it in the current HTML.

### 19.2 — Implementation Approach

Two options for reordering:

**Option A: Drag and Drop** — The more intuitive approach. User grabs a layer row and drags it up or down in the list. A visual placeholder shows where the layer will land. On drop, send `reorderLayer(index, newIndex)` to AE.

Implementation complexity: High. Requires pointer capture, drag visual feedback (ghost element or placeholder), scroll-while-draging behavior (auto-scroll the layer list when dragging near the top/bottom edge), and proper index calculation.

**Option B: Move Up/Move Down Buttons** — Simpler. Two small arrow buttons (↑ and ↓) in the expanded layer view. Clicking ↑ moves the layer up one position, ↓ moves it down. Each click sends `reorderLayer`.

Implementation complexity: Low. Two buttons, two handlers, each calls `reorderLayer` with `index - 1` or `index + 1` (clamped to valid range), then refreshes the layer list.

**Recommendation:** Implement **Option B** first. It's simpler, more reliable over RDP (where drag events can be laggy), and works well on touch devices. If there's demand later, add drag-and-drop as an enhancement.

**Move buttons placement:** In the expanded layer view, next to the action buttons (Split, Duplicate, Delete). Add "↑" and "↓" buttons. These should be visually distinct from the destructive actions (Split, Delete) — place them before Split in the button order, separated by a subtle divider.

### 21.1 — Effect Search Debouncing

As noted in the initial analysis, the effect search queries AE on every keystroke. This must be debounced.

**Implementation:** On the search input's `input` event, clear any existing debounce timer, set a new timer for 250ms. When the timer fires, execute the search. This means rapid typing only triggers one search query (250ms after the last keystroke).

```
  clearTimeout(searchTimer);
  searchTimer = setTimeout(() => searchEffects(searchInput.value), 250);
});
```

## 24. TRANSFORM PANEL — COMPLETE ARCHITECTURE OVERVIEW

### 24.1 — Current State Assessment

The transform panel currently renders every transform property returned by `getLayerProperties` using a generic recursive renderer. This means position, anchor point, scale, rotation, opacity, and any nested property groups all render with the same scrub dial template. While this recursive approach is architecturally flexible (it handles arbitrary property trees), it produces a flat, undifferentiated UI where every property looks identical regardless of its semantic importance or usage frequency.

The result is that commonly adjusted properties (position, rotation, opacity) are visually indistinguishable from rarely-touched properties (anchor point Z component, auto-orient settings). The user has no visual hierarchy guiding them toward the controls they need most.

### 24.2 — Redesign Philosophy

The transform panel redesign follows a **progressive disclosure** model borrowed from professional color grading interfaces (DaVinci Resolve, Baselight) and motion design tools (Cavalry, Rive):

- **Tier 1 (always visible):** Position X/Y, Rotation, Scale, Opacity — the "big four" that every AE artist touches constantly
- **Tier 2 (one click away):** Anchor Point, Per-component Scale (X/Y/Z), Position Z (for 3D layers), Auto-Orient
- **Tier 3 (nested):** All remaining transform sub-properties, expression controls, spatial interpolation

This tiering means the transform panel opens showing the four most important controls prominently. Everything else is accessible through expandable groups but doesn't consume visual real estate by default.

## 25. SCRUB DIAL — COMPLETE REDESIGN

### 25.1 — Current Problems (Detailed)

The existing scrub dial implementation has several interconnected issues that collectively make transform manipulation unreliable:

**Problem A: Static sensitivity values.** The current sensitivity constants (0.5 normal, 0.05 shift, 5 ctrl) are hardcoded and don't adapt to the property being manipulated. For opacity (range 0-100), a 0.5 multiplier means a 100px mouse drag changes the value by 50 — half the entire range in a single gesture. For position on a 4K comp (range 0-3840), the same 0.5 multiplier means a 100px drag changes by 50 — barely 1.3% of the range. The sensitivity should be proportional to the property's meaningful range.

**Problem B: No value clamping.** If the user drags a scrub dial for opacity beyond 100 or below 0, the value goes out of range. AE will clamp it internally, but the UI display shows an out-of-range value until the next property refresh. This creates a disconnection between what the UI shows and what AE actually applied.

**Problem C: Tick marks are decorative noise.** The tick marks translate by the pixel delta of the drag, giving the illusion of measurement. But they don't represent any meaningful unit. They add visual complexity without informational value. On RDP sessions, the tick mark rendering adds paint cost to every pointer move event, contributing to input lag.

**Problem D: No visual feedback during drag.** The user drags the dial and watches a number change. There's no spatial/visual representation of the value within its range. Professional control surfaces use linear faders (value mapped to position along a track) or rotary knobs (value mapped to angle). The current implementation has neither.

**Problem E: The startValue capture can produce NaN.** As noted in the initial analysis, if `track.dataset.value` is not properly populated during property rendering, `parseFloat()` returns NaN. All subsequent arithmetic produces NaN, and the dial becomes unresponsive with no error indication.

### 25.2 — New Scrub Dial Component Specification

The new scrub dial is a **horizontal linear fader** — a thin track with a draggable handle. The handle position represents the current value within the property's range. This provides both a manipulation surface AND a visual value indicator.

**Visual structure:**

```
Label  [==========●========================]  Value Input [kf]
       min                                    max
```

- **Label:** Property name (e.g., "X", "Y", "Rotation", "Opacity") — left-aligned, fixed width
- **Track:** A horizontal bar, 100% of available width minus label and value input. Thin (3px height), with a filled portion from the minimum to the handle position using accent color.
- **Handle:** A circular or vertical-bar element positioned at the value's proportional location within the track. Size: 12px diameter circle, or 3px wide × 16px tall vertical bar. Slightly larger than the track to be easily grabbable. On hover: subtle scale increase (1.2x) via CSS transform.
- **Value Input:** A number input field to the right of the track. Shows the current numeric value. Editable directly — the user can type a value and press Enter to set it. Updates in real-time during scrub drag. Width: 55-65px depending on expected value magnitude.
- **Keyframe Button:** A small diamond icon (◇ when no keyframe at current time, ◆ when keyframe exists) to the right of the value input. Clicking it toggles a keyframe at the current time for this property.

### 25.3 — Handle Position Calculation

The handle position is mapped from the property value to the track width:

```
handlePosition = ((currentValue - minValue) / (maxValue - minValue)) * trackWidth
```

For properties with known ranges:
- **Opacity:** min=0, max=100
- **Rotation:** min=-∞, max=+∞ (but practically, -3600 to 3600 covers 10 full turns in each direction)
- **Scale:** min=0, max=1000 (percentage, 0% to 1000%)
- **Position:** min depends on comp size, max depends on comp size. For a 1920×1080 comp: X range approximately -2000 to 4000, Y range approximately -2000 to 3000. These are soft bounds — position is technically unbounded, but the practical range is a few multiples of the comp size.
- **Anchor Point:** Same practical range as position

For properties where min/max are not well-defined (position, anchor point, custom expression-driven values), use a **dynamic range** approach:
1. Set the initial range to [currentValue - 500, currentValue + 500]
2. If the user drags the handle to both extremes during a single interaction, dynamically expand the range by 2x in the exceeded direction
3. When the property value is set via the number input (direct entry), re-center the range around the new value

For properties with well-defined ranges (opacity, scale as percentage), use the hard min/max. The handle hits the track edge at min/max, providing tactile feedback that the limit has been reached.

### 25.4 — Dynamic Sensitivity Calculation

Replace the static sensitivity constants with a range-proportional calculation:

```
effectiveSensitivity = (maxValue - minValue) / trackPixelWidth * pointerDeltaPixels
```

This means: dragging the handle across the full track width changes the value from min to max. The sensitivity is inherently correct for the property's range.

**Modifier keys:**
- **No modifier:** Full range mapped to full track width (1:1 mapping as described above)
- **Shift (fine):** 10x reduction. The user needs to drag 10x further to achieve the same value change. Equivalent to zooming into the range by 10x.
- **Ctrl (coarse):** 10x amplification. Tiny movements produce large value changes. Useful for quickly sweeping through large ranges.

**Implementation in pointer move handler:**

```
delta = pointerCurrentX - pointerStartX  // pixel delta
rawChange = (delta / trackPixelWidth) * (maxValue - minValue)

if (shiftKey) rawChange *= 0.1
if (ctrlKey) rawChange *= 10

newValue = startValue + rawChange
newValue = clamp(newValue, minValue, maxValue)  // for bounded properties
```

For unbounded properties (position), skip the clamp, but visually constrain the handle to the track edges. If the value exceeds the visual range, the handle sits at the edge and the number input shows the actual value.

### 25.5 — Handle Visual During Drag

While dragging:
- Handle scales up slightly (CSS class toggle: `.scrub-handle:active { transform: scale(1.3); }`)
- The filled portion of the track (from min to handle) is highlighted with accent color
- The unfilled portion (from handle to max) is dimmed
- The value input updates in real-time showing the changing value
- The value input text could optionally change color (accent) during drag to indicate active manipulation, reverting to normal text color on pointer up

### 25.6 — Number Input Direct Entry

The value input field to the right of each scrub dial allows direct numeric entry. Behavior:

- **On focus (click or tab):** Select all text in the input for easy replacement
- **On Enter:** Parse the value, validate it (is it a number? is it within bounds for bounded properties?), send to AE via `setPropertyValue`, update the handle position
- **On blur (clicking elsewhere):** Same as Enter — commit the value
- **On Escape:** Revert to the previous value, exit edit mode
- **On Arrow Up/Down while focused:** Increment/decrement by a small step (1 for integers, 0.1 for frame rate, etc.). With Shift held: larger step (10). This provides keyboard-based scrubbing without the dial.

**Validation rules:**
- Opacity: clamp to [0, 100]
- Scale: clamp to [0, 1000] (or warn if above 100 — "Scale above 100% — intended?")
- Rotation: no clamp (any value is valid, including multiple rotations like 720°)
- Position: no clamp (any value is valid)
- Frame rate: clamp to [0.001, 999]
- Duration: clamp to [0.01, 10800]

### 25.7 — Tick Marks — Removal and Replacement

**Remove the existing tick mark system entirely.** The scrub dial track with handle position provides all necessary visual feedback. Tick marks added visual noise without information.

Replace with: a subtle **value range indicator**. For bounded properties, small min/max labels at the track endpoints (e.g., "0" and "100" for opacity). These labels use the muted text color and very small font size (9-10px). They're barely noticeable but provide context for the handle position.

### 25.8 — Keyframe Button State

The keyframe diamond icon next to each property must reflect whether a keyframe exists at the current composition time for that specific property:

- **No keyframe at current time:** ◇ (outline diamond), muted color
- **Keyframe at current time:** ◆ (filled diamond), accent color
- **Keyframe between current time and adjacent keyframes:** The diamond could show a small dot or half-filled state (nice-to-have, not essential)

**How to determine keyframe existence:** The `getLayerProperties` response from AE may include keyframe timing data per property. If it does, check if the current composition time falls on any keyframe time (with a small epsilon tolerance for floating-point comparison — `Math.abs(keyTime - currentTime) < 0.001`). If the response doesn't include keyframe data, add a separate lightweight command: `getKeyframeTimes(layerIndex, propertyPath)` that returns an array of keyframe times. Cache this data and only refresh it when the property tree is reloaded or when the playhead moves.

**Click behavior:**
- If no keyframe at current time: send `setKeyframe` command → icon fills → show subtle "pop" animation (scale 1→1.3→1, 150ms)
- If keyframe exists at current time: send `removeKeyframeAtTime` command → icon empties

## 26. TIER 1 CONTROLS — DETAILED SPECIFICATION

### 26.1 — Position Control

**Layout:**
```
Position    X [====●===========] 960 [kf]   Y [====●===========] 540 [kf]
```

**Two scrub dials** on separate lines, labeled X and Y. For 3D layers, a third line appears for Z.

**X and Y are rendered as separate controls** rather than a combined 2D pad because:
1. It's easier to make precise single-axis adjustments with a horizontal fader
2. The Trackpad section (described below) handles 2D manipulation
3. It matches the mental model of AE's own position property display

**Value display:** Integer pixels by default (e.g., 960). If the value has a sub-pixel component (common with smooth keyframes or expressions), show one decimal place (e.g., 960.5).

**Position Z (3D layers only):** Appears as a third scrub dial below Y, labeled "Z". Only rendered when the layer has the 3D flag enabled. When the layer is 2D, this line is not rendered (not hidden — removed from DOM entirely to avoid confusion).

### 26.2 — Rotation Control

**Layout:**
```
Rotation    [===========●===================] 45.0° [kf]
```

**Single scrub dial** spanning the full property width. The degree symbol (°) is appended to the value display.

**Rotation range:** The scrub dial's visual range should be -360° to +360° by default (two full rotations). The handle maps to this range. If the value exceeds ±360° (e.g., a spinning animation at 720°), the handle saturates at the track edge, and the number input shows the actual value. The user can still drag to increase/decrease beyond the visual range — the handle just stays at the edge while the value continues changing at the same rate.

**Rotation wrapping consideration:** AE does NOT wrap rotation values. A layer rotated 3 times shows 1080°, not 0°. The UI must preserve this. Don't implement modular arithmetic on the displayed value.

**Multi-rotation keyframes:** If the user is creating a spinning animation (e.g., rotation goes from 0° to 720°), the scrub dial is inadequate for this use case because its visual range only covers 720°. For multi-turn rotations, the user should use the number input to type exact values. This is acceptable — the scrub dial handles 95% of rotation use cases, and the number input handles the remaining 5%.

### 26.3 — Scale Control

**Layout:**
```
Scale       [===========●===================] 75% [kf]
```

**Single scrub dial** for uniform scale. The value is displayed as a percentage.

**Scale representation:** AE stores scale as an array [scaleX, scaleY] (or [scaleX, scaleY, scaleZ] for 3D layers), with values typically in the range 0-100 (representing 0%-100%). The UI displays this as a percentage.

**Uniform vs. non-uniform scale detection:**
- If scaleX === scaleY (and scaleZ for 3D): show a single scrub dial labeled "Scale" with the uniform value
- If scaleX !== scaleY: show TWO scrub dials labeled "Scale X" and "Scale Y" (and "Scale Z" for 3D). This is the Tier 2 "Per-Component Scale" expanded view — but the detection happens automatically

**Initial display:** Always show the uniform control first. If the values are non-uniform, display the X component value in the single scrub dial and add a small indicator (e.g., "ⓘ non-uniform") that hints at the per-component expansion below. When the user drags the uniform scrub dial, it sets BOTH X and Y to the same value (effectively making the scale uniform again). This is a deliberate behavior — it's the most common intent when someone grabs the scale control.

### 26.4 — Opacity Control

**Layout:**
```
Opacity     [===========●===================] 75% [kf]
```

**Single scrub dial** with bounded range 0-100. The percentage sign is appended to the value display.

**The handle reaches the left edge at 0% and the right edge at 100%.** This is the simplest, most constrained scrub dial — opacity always has a well-defined range.

**Opacity display during drag:** As the user drags, the value input updates in real-time. Additionally, the layer's opacity in AE changes in real-time (via throttled `setPropertyValue` commands). This gives the user immediate visual feedback in the AE viewport (if they can see it on another monitor or via RDP).

## 27. TIER 2 CONTROLS — EXPANDABLE GROUPS

### 27.1 — Expandable Group Component

Tier 2 controls are organized into collapsible groups. Each group has a **header row** that acts as the toggle:

```
▸ Anchor Point                    [group keyframe button]
```

Clicking the header row toggles the group content:

```
▾ Anchor Point                    [group keyframe button]
  X [====●===========] 0 [kf]
  Y [====●===========] 0 [kf]
  Z [====●===========] 0 [kf]
```

The chevron (▸/▾) rotates on toggle. The group content animates in with `max-height` transition (same pattern as the comp settings panel).

**Group keyframe button:** The `[kf]` button in the group header sets/removes keyframes for ALL properties within the group simultaneously. This is useful for keyframing the entire anchor point at once rather than individually keyframing X, Y, and Z.

### 27.2 — Anchor Point Group

Contains X, Y, and Z (for 3D layers) scrub dials. Same structure as position. The key difference is that anchor point values are relative to the layer's coordinate system, not the composition's. For a 1920×1080 solid, the default anchor point is [960, 540] (center). The scrub dial range should be [0, layerWidth] for X and [0, layerHeight] for Y as defaults, dynamically adjusted based on the layer's dimensions (which the bridge returns).

**UX consideration:** Changing the anchor point can dramatically alter a layer's visual position without changing the position values. This is confusing for new users. Consider adding a small tooltip or info icon next to the Anchor Point group header that, on hover, explains: "The anchor point is the layer's transform origin. Moving it changes where rotation and scale are applied from."

### 27.3 — Per-Component Scale Group

Contains X and Y (and Z for 3D layers) scrub dials for non-uniform scaling. Each dial ranges from 0% to 1000% (generous upper bound — AE allows any scale value, but 1000% covers virtually all practical use cases).

**Link/Unlink toggle:** Next to the group header, a small **link icon** (🔗) that, when active, constrains X and Y to change together (proportional scaling). When inactive (broken link icon 🔗 with a slash), X and Y change independently. Default state: linked.

When linked and the user drags one dial, the other dial's value changes proportionally. For example, if current scale is [80%, 60%] and the user drags X to 100% (a 25% increase), Y should increase by 25% to 75%. This maintains the aspect ratio.

When unlinked, each dial operates independently.

**Technical implementation of linked scaling:**
- Store the ratio at the moment of link activation: `scaleRatio = scaleY / scaleX`
- When X changes while linked: `newY = newX * scaleRatio`
- When Y changes while linked: `newX = newY / scaleRatio`
- If the ratio changes externally (e.g., from AE), update the stored ratio on next property refresh

### 27.4 — Auto-Orient Group

AE's auto-orient feature makes a layer automatically rotate to follow its motion path, face the camera, or face a point of interest. The group contains:

- **Auto-Orient dropdown:** Options: `Off | Along Path | Towards Camera | Towards Point of Interest`
- Maps to `layer.autoOrient` with values: `AutoOrientType.NO_AUTO_ORIENT`, `AutoOrientType.ALONG_PATH`, `AutoOrientType.CAMERA_OR_POINT_OF_INTEREST`

The dropdown is the only control in this group. No scrub dials. It's a simple configuration dropdown. When "Off" is selected, no further controls appear. When "Along Path" is selected, a brief note appears: "Layer will rotate to follow its motion path." When "Towards Camera" is selected, a note: "Layer will face the active camera."

### 27.5 — Time & Speed Group

Contains:

- **Time Stretch:** Number input showing the current time stretch percentage. 100% = normal speed, 200% = half speed, 50% = double speed. Input range: -10000% to 10000% (AE supports extreme stretch values). The sign indicates direction: positive = forward, negative = reverse.
  
- **Reverse toggle:** A toggle switch that, when activated, sets the time stretch to negative its current value. This is a convenience shortcut for the common operation of reversing a layer. If time stretch is 100% and reverse is toggled on, time stretch becomes -100%. If toggled off, it returns to 100%.

- **Frame Blending dropdown:** Options: `Off | Frame Mix | Pixel Motion`. Maps to `layer.frameBlendingType`.

- **Loop Expression dropdown:** Options: `None | Cycle | Pingpong | Offset | Continue`. When a loop type is selected, a loopOut expression is applied to the layer's time remapping or the selected property. When "None" is selected, any existing loopOut expression is removed.

**Loop expression implementation detail:** The existing codebase applies loop expressions to position specifically. This should be generalized — the loop expression should be applied to the property that is currently selected or focused, not hardcoded to position. If no property is focused, apply to the property that makes the most sense (usually time remapping if enabled, otherwise position). The user should be able to choose which property gets the loop expression — either via a dropdown within this group or by applying it contextually to whichever property is currently active in the scrub dial system.

## 28. TRACKPAD CONTROL SURFACE — COMPLETE REDESIGN

### 28.1 — Current Problems

**Problem A: Stale cached values.** The trackpad reads current property values when a layer is selected and stores them in a `trackpadValue` variable. If the user then adjusts a value via scrub dial, the trackpad's cached value doesn't update. The next trackpad drag starts from the stale value, causing a visible jump.

**Problem B: No visual value representation.** The trackpad is a flat rectangle that accepts drags. It shows the resulting coordinates as text below the surface, but there's no visual indicator of the current property value's position within the trackpad area. The user has no spatial reference point.

**Problem C: 1:1 pixel mapping for position.** Position mode maps screen pixels 1:1 to AE pixels. But the trackpad surface might be 300×200 pixels in the browser while the comp is 3840×2160. A full drag across the trackpad only moves the layer 300px in X and 200px in Y — a tiny fraction of the comp. The mapping feels sluggish and disconnected.

**Problem D: Rotation sensitivity is too low.** `dx * 0.5` means a 100px horizontal drag produces 50° of rotation. This seems reasonable, but on a trackpad surface of 300px, a full-width drag produces 150° — not even half a rotation. For precise rotation work, the user needs very small drags, which are hard to control via RDP.

**Problem E: No scroll/pinch support.** Modern trackpads and touch devices support scroll wheel and pinch gestures. The current implementation only handles pointer drag. Adding scroll-to-rotate or pinch-to-scale would dramatically improve the touch experience.

### 28.2 — Redesigned Trackpad Layout

```
┌──────────────────────────────────────────────┐
│ Mode: [Position] [Rotation] [Scale] [Anchor] │
├──────────────────────────────────────────────┤
│                                              │
│                   ◉                           │ ← crosshair indicator showing current value
│                                              │
│              [large drag surface]             │
│                                              │
│                                              │
├──────────────────────────────────────────────┤
│ X: 960.0   Y: 540.0   Δ: +12.5, -3.0       │
└──────────────────────────────────────────────┘
```

### 28.3 — Crosshair Indicator

A small crosshair (⊕) rendered at the position corresponding to the current property value within the trackpad's coordinate space. This gives the user a **visual reference point** for where the current value sits.

**For Position mode:** The crosshair maps the layer's position to the trackpad surface. If the comp is 1920×1080 and the layer is at [960, 540] (center), the crosshair sits at the center of the trackpad. If the layer moves to [1920, 0] (top-right corner), the crosshair moves to the top-right of the trackpad.

**Mapping formula:**
```
crosshairX = (layerX / compWidth) * trackpadWidth
crosshairY = (layerY / compHeight) * trackpadHeight
```

**For Rotation mode:** The crosshair is replaced by a **circular indicator** or simply centered (since rotation is a single-axis value, not a 2D position). The current rotation angle could be shown as a line from the center at the current angle, like a clock hand.

**For Scale mode:** The crosshair represents uniform scale. Center = 0% scale, edge = 200% scale (or some sensible range). Or, if per-component scale is active, X axis maps to scaleX, Y axis maps to scaleY — the crosshair shows the current scale relationship.

**For Anchor Point mode:** Same mapping as Position mode, but relative to the layer's coordinate system instead of the comp's.

### 28.4 — Scaled Position Mapping

Replace the 1:1 pixel mapping with a comp-proportional mapping:

```
positionDeltaX = (pointerDeltaX / trackpadWidth) * compWidth * sensitivityMultiplier
positionDeltaY = (pointerDeltaY / trackpadHeight) * compHeight * sensitivityMultiplier
```

The `sensitivityMultiplier` defaults to `1.0`, meaning a full drag across the trackpad moves the layer across the full comp. The user can optionally adjust this (via a small slider or predefined presets: 0.25x, 0.5x, 1x, 2x, 4x) for finer or coarser control.

With this mapping on a 1920×1080 comp and a 300×200 trackpad:
- A 30px drag (10% of trackpad width) = 192px of position change
- A full-width drag = 1920px (full comp width)

This feels much more natural and responsive.

### 28.5 — Rotation Sensitivity Adjustment

Replace `dx * 0.5` with:

```
rotationDelta = (dx / trackpadWidth) * 360 * sensitivityMultiplier
```

This means a full-width drag = 360° (one full rotation). A 10% drag = 36°. This is a much more intuitive mapping. The `sensitivityMultiplier` defaults to 1.0.

### 28.6 — Scale Sensitivity Adjustment

Replace the current `* 0.2` with:

```
scaleDeltaX = (dx / trackpadWidth) * 200 * sensitivityMultiplier
scaleDeltaY = (dy / trackpadHeight) * 200 * sensitivityMultiplier
```

A full-width drag = 200% scale change. Default sensitivity = 1.0. This allows the user to go from 0% to 200% scale with a full drag, which covers most practical range.

### 28.7 — Delta Display

Below the trackpad surface, show the **current absolute values** AND the **delta** (change) since the drag started:

```
X: 960.0   Y: 540.0   Δ: +12.5, -3.0
```

The absolute values update in real-time during drag. The delta shows how much the value has changed since pointer down. On pointer up, the delta resets to zero. This helps the user make precise relative adjustments (e.g., "I want to move this exactly 50 pixels right").

### 28.8 — Scroll Wheel Support

Attach a `wheel` event listener to the trackpad surface:

- **Rotation mode:** Scroll up = rotate clockwise, scroll down = counter-clockwise. Each scroll "notch" = 5° (configurable). With Shift held: 1° per notch (fine). With Ctrl held: 15° per notch (coarse).
- **Scale mode:** Scroll up = scale up, scroll down = scale down. Each notch = 5% change. Shift = 1%, Ctrl = 15%.
- **Position mode:** Scroll up/down = Y axis movement, with Shift+scroll = X axis movement. Each notch = 10px (scaled to comp size as described above).

**Implementation:**
```
trackpadElement.addEventListener('wheel', (e) => {
  e.preventDefault();
  const direction = e.deltaY < 0 ? 1 : -1;
  const step = e.shiftKey ? fineStep : e.ctrlKey ? coarseStep : normalStep;
  const delta = direction * step;
  // Apply delta to the appropriate axis based on current mode
  // Send update to AE
}, { passive: false });
```

The `{ passive: false }` option is required because we need to call `e.preventDefault()` to prevent the page from scrolling.

### 28.9 — Undo Group Integrity

The existing undo group wrapping (`beginUndoGroup` on pointer down, `endUndoGroup` on pointer up) is correct and must be preserved. Additionally:

- For scroll wheel operations, each scroll event should also be wrapped in an undo group. However, rapidly scrolling generates many events. To prevent the undo stack from filling with hundreds of micro-adjustments, debounce the undo group: `beginUndoGroup` on the first scroll event, `endUndoGroup` 300ms after the last scroll event. This groups rapid scrolling into a single undoable action.

- For the crosshair indicator updates during drag, these should NOT send undo groups — they're visual only. The undo group is created by the actual value-setting commands.

### 28.10 — Pointer Capture — Correctness

The existing pointer capture implementation (`element.setPointerCapture(e.pointerId)`) is correct for ensuring that pointer events continue to fire on the trackpad even when the pointer moves outside the element. This must be preserved.

**One refinement:** On pointer up, release the capture with `element.releasePointerCapture(e.pointerId)`. The existing code may or may not do this explicitly — the capture is automatically released on pointer up in most browsers, but explicit release is best practice for clarity and cross-browser reliability.

### 28.11 — RAF Throttling — Refinement

The existing `requestAnimationFrame` throttling for value updates is correct. One refinement: during the RAF callback, check if the value has actually changed since the last send. If the pointer moved but the calculated value is the same (e.g., due to rounding or very small movements), skip the AE command. This prevents redundant commands:

```
function rafCallback() {
  if (pendingValue !== lastSentValue) {
    send({command: 'setValueAtTime', args: {value: pendingValue, ...}});
    lastSentValue = pendingValue;
  }
  if (isDragging) requestAnimationFrame(rafCallback);
}
```

## 29. PROPERTY TYPE HANDLING — COMPREHENSIVE GUIDE

### 29.1 — Number Properties

The most common property type. Rendered as a single scrub dial + value input + keyframe button.

**Examples:** Rotation, Opacity, individual position components, effect slider values (blur amount, etc.)

**Bridge response shape:** `{type: 'number', value: 45.0, name: 'Rotation', ...}`

**Rendering:** One scrub dial. Handle position = `(value - min) / (max - min) * trackWidth`. For unknown ranges, use dynamic range centering on current value (section 25.3).

**Setting value:** `send({command: 'setPropertyValue', args: {path: propPath, value: newValue, layerIndex: idx}})`

### 29.2 — Array Properties (2D/3D Position, Scale, Anchor)

Properties that return an array of numbers. Each component is rendered as its own scrub dial.

**Examples:** Position [x, y] or [x, y, z], Scale [x, y] or [x, y, z], Anchor Point [x, y, z]

**Bridge response shape:** `{type: 'array', value: [960, 540, 0], name: 'Position', ...}`

**Rendering:** N scrub dials, one per component, labeled X, Y, Z respectively. Each dial's value input shows the individual component.

**Setting value:** When one component changes, construct the full array with the new value at the changed index and send it. Example: if Y changes from 540 to 600, send `{path: 'ADBE Position', value: [960, 600, 0]}`. This requires reading the current values of the OTHER components at the time of the send (from the scrub dial's current state, not from the initial property read, because they may have changed).

**Edge case:** If the array length changes (e.g., layer goes from 2D to 3D, adding a Z component), the scrub dial set must be re-rendered. Detect this during property refresh and rebuild the component dials.

### 29.3 — Color Properties

Properties that return color values.

**Examples:** Effect color controls, shape fill color, shape stroke color, solid color

**Bridge response shape:** `{type: 'color', value: [1, 0, 0, 1], name: 'Color', ...}` (RGBA, values 0-1)

**Rendering:** A color swatch (square, 28×28px) showing the current color, plus a native `<input type="color">` for picking. The native color picker doesn't support alpha, so alpha is handled separately as a scrub dial labeled "A" or "Alpha".

**Swatch click:** Opens the native color picker (via the hidden input's `.click()` method).

**Color picker change:** Converts the hex color from the picker to RGBA (0-1 range), combines with the current alpha value, sends to AE.

**Alpha scrub dial:** Rendered below the color swatch. Range 0-1. Displayed as 0%-100%.

**Value input:** Show the hex code (e.g., "#FF3300") as a read-only label next to the swatch. Optionally, make it editable for users who prefer typing hex values.

### 29.4 — Boolean Properties

Properties that return true/false.

**Examples:** Layer enabled, layer locked, layer solo, layer shy, motion blur, 3D layer flag, various effect toggles

**Rendering:** A toggle switch (same component as comp settings toggles). On toggle, send the new boolean value immediately.

**In the transform panel context:** These are mostly handled by the flag toggles in the layer list (section 18), not in the scrub dial system. But within effect properties, booleans render as toggle switches.

### 29.5 — String Properties

Properties that return text strings.

**Examples:** Source text content, effect menu selections

**Rendering:** For short strings (under 20 characters), a text input. For longer strings, a textarea.

**For effect menu properties** (dropdown selects within effects like "Direction" in Radial Blur being "Directional/Angular"), render as a `<select>` dropdown with the available options. The bridge would need to return the available options for menu-type properties — if it doesn't, render as a generic text input.

### 29.6 — MarkerValue Properties

AE markers (composition markers, layer markers) have complex data: name, comment, chapter, URL, frame target, duration, cue point name.

**Rendering:** These are not typical scrub dial properties. They should be rendered as **expandable cards** within the property tree, showing the marker's time, name, and comment. Editing is limited to name and comment (text inputs) and time (number input). This is a specialized case that doesn't use the scrub dial template.

### 29.7 — ShapeValue Properties

Path shape data (Bezier paths for shape layers). These are complex data structures containing vertices, in-tangents, out-tangents, and closed/open state.

**Rendering:** These cannot be edited with scrub dials. Display them as read-only information: "Path (12 vertices)" with an expand option showing the raw vertex data. Path editing is a canvas-based feature that is out of scope for the scrub dial system — it would require a dedicated path editor component (future enhancement).

## 30. TIMELINE BAR — DETAILED IMPLEMENTATION

### 30.1 — Component Structure

The timeline bar sits between the comp selector toolbar and the tab strip. It is **always visible** regardless of which tab is active. It contains:

```
[⏮] [◀] [▶/⏸] [▶] [⏭] | 00:00:15:12 | [========playhead scrubber========] | F: 45 / 120
```

### 30.2 — Transport Buttons

Five buttons in a button group:

| Button | Icon | AE Command | Description |
|--------|------|------------|-------------|
| Go to Start | ⏮ | `setCurrentTime(0)` | Moves playhead to time 0 |
| Previous Frame | ◀ | `setCurrentTime(currentTime - frameDuration)` | Steps back one frame |
| Play/Pause | ▶/⏸ | `playPause` | Toggles playback |
| Next Frame | ▶ | `setCurrentTime(currentTime + frameDuration)` | Steps forward one frame |
| Go to End | ⏭ | `setCurrentTime(totalDuration)` | Moves playhead to comp end |

**Frame duration calculation:** `1 / frameRate`. For 30fps: `1/30 ≈ 0.03333...` seconds per frame. This must be calculated to sufficient decimal precision to avoid drift over many frames. Use the raw fraction rather than a rounded decimal.

**Button sizing:** Each button is a square with minimum 36×36px (44×44 on touch). Icon centered within. Buttons are grouped with no gap between them (shared border or touching backgrounds), forming a unified control strip.

**Active state:** Only Play/Pause has an active state. When playing, the button shows pause icon and has an accent-tinted background.

**Repeat behavior for frame stepping:** If the user holds down Previous Frame or Next Frame button, it should repeat. Implement with `pointerdown` starting an interval (initial delay 400ms, then repeat every 100ms). On `pointerup`, clear the interval. This allows scrubbing frame-by-frame by holding the button — a common workflow for frame-accurate positioning.

### 30.3 — Timecode Display

**Format:** `HH:MM:SS:FF`

- HH: Hours (00-99)
- MM: Minutes (00-59)
- SS: Seconds (00-59)
- FF: Frames (00 to frameRate-1)

**Calculation from AE time (in seconds):**
```
totalFrames = Math.floor(time * frameRate);
hours = Math.floor(totalFrames / (frameRate * 3600));
minutes = Math.floor((totalFrames % (frameRate * 3600)) / (frameRate * 60));
seconds = Math.floor((totalFrames % (frameRate * 60)) / frameRate);
frames = totalFrames % frameRate;
```

**Display:** Monospace font (critical — non-monospace fonts cause the display to jump as digits change width). Font size: slightly larger than surrounding text (14-16px). Color: primary text color on a slightly different background shade to distinguish it as a readout.

**Update frequency:** Every 500ms during playback (existing polling). Immediately after frame step commands. Immediately after playhead scrubber interaction.

**Editable (nice-to-have):** Making the timecode display editable (click to enter a specific timecode, press Enter to jump there) is a power-user feature. Implement as a future enhancement, not in the initial build.

### 30.4 — Playhead Scrubber

**Visual:** A horizontal track representing the full duration of the composition. The track background is a muted color. The portion from the start to the current time is slightly highlighted (accent color at low opacity). A vertical handle (3px wide, full track height) marks the current position.

**Track dimensions:** Full available width minus the transport buttons and timecode display. Minimum width: 200px. Height: 8px visual, 32px hit area (invisible padding for touch targets).

**Handle:** A small triangle or circle at the top of the track, centered on the current time position. This is the draggable element. On hover: handle extends or enlarges slightly. On drag: handle follows pointer.

**Interaction model:**
1. `pointerdown` on track or handle: capture pointer, begin scrub
2. `pointermove`: calculate time from pointer position, update handle visually, update timecode display, send `setCurrentTime` to AE (throttled via RAF to 60fps max)
3. `pointerup`: release capture, final time already committed

**Time calculation from pointer position:**
```
fraction = (pointerX - trackRect.left) / trackRect.width
fraction = clamp(fraction, 0, 1)
time = fraction * totalDuration
```

**Performance consideration during scrub:** Sending `setCurrentTime` on every RAF frame during a scrub means up to 60 commands per second. AE should handle this fine since it's just moving the playhead (no rendering). But if performance issues arise, reduce to 30fps or implement command coalescing (only send if the time value has changed by at least one frame duration).

**Keyframe markers on the scrubber (enhancement):** If keyframe data is available for the currently selected property, small dots or ticks on the scrubber track at keyframe times. These are purely visual — not interactive. They help the user align the playhead with keyframes. Implement only after the core scrubber works correctly.

### 30.5 — Frame Counter

**Format:** `F: 45 / 120` (current frame / total frames)

**Calculation:**
```
currentFrame = Math.floor(currentTime * frameRate)
totalFrames = Math.floor(compDuration * frameRate)
```

**Display:** Small, monospace font, muted text color. Positioned to the right of the scrubber.

### 30.6 — Transport State Management

**Polling lifecycle:**
- On AE connection (status green): start transport polling
- Poll every 500ms via `getCurrentTime` command
- Update timecode display, frame counter, and playhead scrubber handle position from response
- On AE disconnect (status red/amber): stop polling

**Playing state detection:**
- After sending `playPause`, toggle `state.isPlaying`
- During polling, if two consecutive polls return the same time value while `state.isPlaying` is true, assume AE has stopped playing (either hit end of comp, or user stopped in AE directly). Set `state.isPlaying = false` and update the play/pause button icon.

**Edge case:** AE may return times that are slightly imprecise due to floating-point arithmetic. Use an epsilon comparison: `Math.abs(time1 - time2) < frameDuration / 2` to consider two times as "the same frame."

## 33. PANEL SCROLL BEHAVIOR

### 33.1 — Fixed Header, Scrollable Content

The transform panel (and all tab content panels) must scroll independently of the page layout. The header bar, comp settings, comp selector, timeline bar, and tab strip are all **fixed position** relative to the viewport. Only the active panel content scrolls.

**CSS approach:**
```
.panel-container {
  flex: 1;
  overflow-y: auto;
  overflow-x: hidden;
  -webkit-overflow-scrolling: touch;  /* smooth scrolling on iOS */
}
```

The parent flex container uses `height: 100vh; display: flex; flex-direction: column;`. The header, settings, toolbar, timeline, and tabs are all `flex-shrink: 0`. The panel container is `flex: 1; overflow-y: auto;`.

### 33.2 — Scroll Position Persistence

When switching tabs, the scroll position of the previous tab should be remembered. If the user scrolls deep into the transform panel, switches to Effects, then switches back to Transform, the scroll position should be restored to where they left off.

**Implementation:** Store each panel's `scrollTop` value in a Map keyed by tab name. On tab switch: save current panel's `scrollTop`, restore new panel's `scrollTop`.

```
const scrollPositions = new Map();

function switchTab(tabName) {
  // Save current scroll position
  const currentPanel = document.querySelector('.panel.active');
  if (currentPanel) {
    scrollPositions.set(state.activeTab, currentPanel.scrollTop);
  }
  // Switch panel
  // ...
  // Restore new scroll position
  const newPanel = document.querySelector('.panel.active');
  if (newPanel && scrollPositions.has(tabName)) {
    newPanel.scrollTop = scrollPositions.get(tabName);
  }
}
```

## 34. PERFORMANCE — TRANSFORM OPERATIONS

### 34.1 — Command Throttling Summary

All interactive controls (scrub dials, trackpad, playhead scrubber) must throttle their AE commands to prevent flooding. The existing RAF throttling is the correct approach. Summary of all throttled operations:

| Control | Throttle Method | Max Frequency | Notes |
|---------|----------------|---------------|-------|
| Scrub Dial (drag) | RAF | ~60fps | Only send if value changed |
| Trackpad (drag) | RAF | ~60fps | Existing implementation, add value-change check |
| Playhead Scrubber | RAF | ~60fps | May reduce to 30fps if needed |
| Scroll Wheel | Debounce | 300ms | Groups rapid scrolls into one undo group |
| Frame Step Buttons | Direct (on click) + Interval (on hold) | 100ms repeat | No throttle needed for single clicks |
| Number Input | On Enter/blur only | N/A | One command per edit |

### 34.2 — Property Refresh Avoidance During Interaction

During any active manipulation (scrub dial drag, trackpad drag, playhead scrub), the periodic `loadLayerProperties()` and transport polling must not interfere. If a property refresh fires while the user is mid-drag, it could overwrite the in-progress value with the slightly-stale AE state (due to command latency).

**Solution:** Set a flag `isInteracting = true` on pointer down for any scrub dial, trackpad, or scrubber. Clear it on pointer up. While `isInteracting` is true, skip property tree refreshes. Transport polling (getCurrentTime) should continue since it doesn't affect property values.

## 35. EFFECTS PANEL — COMPLETE SPECIFICATION

### 35.1 — Current State Assessment

The effects panel currently has two sections: an active effect list (expandable per effect with sub-properties rendered via the generic scrub group renderer) and an effect search input that queries AE's `getEffectsList`. The core architecture is sound — effects are read from the `ADBE Effect Parade` property group and rendered recursively. However, several functional gaps degrade the experience:

**Gap A: No effect reordering.** AE allows users to drag effects up and down in the effect stack, and the order matters (effects apply top-to-bottom). The current UI has no mechanism for reordering.

**Gap B: Effect search results lack context.** Results show effect names but not their category/folder. A user searching "blur" might see Gaussian Blur, Fast Blur, Camera Lens Blur, Radial Blur, Directional Blur, and CC Radial Blur — but without category information, it's hard to distinguish them or understand which is which.

**Gap C: Effect removal doesn't refresh or confirm.** Clicking the remove button on an effect deletes it immediately. For complex effects with many adjusted parameters, accidental deletion loses all that work.

**Gap D: No effect bypass/toggle.** AE allows toggling an effect on/off without removing it (the effect's "bypass" flag). The current UI lacks this toggle.

**Gap E: Generic property rendering doesn't account for effect-specific controls.** Every effect property renders as the same scrub dial, regardless of whether it's a slider, angle control, color picker, checkbox, or dropdown menu. The rendering should adapt based on the property's semantic type.

### 35.2 — Revised Effects Panel Layout

```
┌─────────────────────────────────────────────────┐
│ [🔍 Search effects...]                          │
│ [results dropdown appears here when searching]   │
├─────────────────────────────────────────────────┤
│                                                 │
```

### 35.3 — Effect Card Structure

Each effect is rendered as a **card** — a visually distinct container with a subtle border, rounded corners, and a surface background color slightly different from the panel background. This creates clear visual boundaries between effects, especially important when multiple effects are expanded simultaneously.

**Card header row** contains (left to right):
- **Effect name:** Bold, primary text color. This is the display name returned by AE (e.g., "Gaussian Blur", not the match name "ADBE Gaussian Blur 2").
- **Bypass toggle (eye icon):** 👁 when active, crossed-out eye when bypassed. Clicking toggles the effect's bypass state. When bypassed, the entire card dims (opacity 0.4) and the effect properties become non-interactive (read-only appearance). Command: `setLayerProperty` targeting the effect's `bypass` or `enabled` property.
- **Reorder arrows (↑↓):** Two small arrow buttons. Up moves the effect higher in the stack (earlier in processing order), down moves it lower. These map to a `reorderEffect` command or `moveEffect` within the effect parade. Each arrow click sends: `send({command: 'reorderEffect', args: {layerIndex, effectIndex, direction: 'up'|'down'}})`. After reorder, refresh the effect list to reflect the new order.
- **Remove button (×):** Red-tinted on hover, not by default. On click: **show a confirmation** before deletion. The confirmation should be a simple inline message within the card: "Remove [effect name]? [Yes] [No]" rather than a browser `confirm()` dialog (which is ugly and blocks the UI). On Yes: send `removeEffect` command, then refresh effects list. On No: dismiss the confirmation.

**Card body** contains the effect's properties, rendered when the card is expanded. Clicking the card header (anywhere except the bypass toggle, reorder arrows, and remove button) toggles expand/collapse.

### 35.4 — Effect Property Rendering — Type-Aware

The generic recursive property renderer works for basic display, but effects contain diverse property types that benefit from specialized rendering. The property type should be detected from the bridge response and mapped to the appropriate control:

**Slider properties** (most common in effects): Properties that represent a numeric range with a known minimum and maximum. AE's effect slider controls return a `matchName` containing "Slider" or the property has `hasMin`/`hasMax` flags. Render as a scrub dial with the effect's defined min/max as bounds. Example: Gaussian Blur's "Blurriness" has a min of 0 and effectively no max — use the scrub dial with min=0, dynamic max.

**Angle properties**: Properties that represent rotation angles. AE's angle controls return a `matchName` containing "Angle" or the property value cycles at 360°. Render as a scrub dial with degree display (° suffix) and a range of -360 to 360 visually (but allow values beyond this range via the number input).

**Color properties**: Properties that return RGBA arrays. Render as color swatch + native color picker + alpha scrub dial, identical to the transform panel color handling (section 29.3).

**Checkbox/Boolean properties**: AE effect checkboxes. Render as a toggle switch. The match name often contains "Checkbox". On toggle, send the boolean value immediately.

**Dropdown/Menu properties**: AE effect popup menus. The match name often contains "Popup" or the property has an `enumNames` array listing the available options. Render as a `<select>` dropdown with the available options. On change, send the selected index or value.

**Point properties**: Effect controls that specify a 2D point (like the center of Radial Blur or the start/end of Linear Wipe). Render as two scrub dials (X, Y) or as a 2D input pair. Ideally, these would be linked to the Trackpad surface — when a point property is focused, the trackpad temporarily enters "Point Edit" mode and allows direct 2D manipulation of that effect's point control.

**Multi-Dimension properties**: Some effects have properties that are multi-dimensional (like "Blur Dimensions" in Gaussian Blur offering "Horizontal and Vertical" vs "Horizontal" vs "Vertical"). These are typically menu/dropdown types.

### 35.5 — Effect Search — Complete Implementation

**Search input:** A text input at the top of the effects panel. Placeholder text: "Search effects...". The input has a search icon (🔍) as a prefix icon (positioned absolutely within the input).

**Debounce:** 250ms after the last keystroke before firing the search query. This prevents flooding AE with search commands during fast typing.

**Minimum query length:** 2 characters. Below 2 characters, don't search and hide any visible results.

**Search execution:** Send `getEffectsList` command to AE. The bridge returns an array of available effects with their display names and match names. Filter the results client-side based on the query string (case-insensitive substring match against the display name). This is more efficient than sending the query string to AE and having it filter — AE's `getEffectsList` likely returns the full list regardless, so filtering client-side avoids an extra bridge round-trip for each filtered result.

**Results dropdown:**
- `position: absolute; z-index: 60;` below the search input
- `max-height: 200px; overflow-y: auto;` to handle long result lists
- Each result is a clickable row showing the effect display name and its category (if available from the bridge response)
- On hover: highlight background
- On click: send `addEffect` command with the effect's match name, close dropdown, clear search input, refresh effect list
- If no results: show "No effects found for '[query]'" as a non-interactive muted text row

**Dismiss behavior:**
- Click outside the dropdown: close
- Escape key: close and clear search input
- Tab switch: close (the effects panel is no longer visible)
- Scroll the effects panel: close (the dropdown position would be wrong relative to the scrolled content)

**Keyboard navigation within results:**
- Arrow Down: highlight next result
- Arrow Up: highlight previous result
- Enter: select highlighted result (same as click)
- Escape: close dropdown

### 35.6 — Add Effect — Alternative Entry Point

Below the effect list (at the bottom of the panel), an "+ Add Effect" button provides an alternative to the search input. Clicking it focuses the search input and opens the results dropdown showing popular/commonly used effects (Gaussian Blur, Curves, Levels, Hue/Saturation, Drop Shadow, etc.) as a starting point. This is useful when the user knows they want to add an effect but isn't sure of the exact name.

### 35.7 — Effect List Refresh After Mutation

After any effect mutation (add, remove, reorder, bypass toggle), the effect list must refresh. Chain `loadEffects()` onto the command's promise resolution:

```
aeCommand('addEffect', {layerIndex, effectMatchName})
  .then(() => loadEffects());
```

When removing an effect, after refresh, the expanded state of the remaining effects should be preserved. Store the expanded state by effect index (or effect match name + index combination), and after re-rendering, re-expand the same effects.

### 35.8 — Effect Property Value Setting

When a scrub dial, toggle, dropdown, or color picker within an effect card is changed, the command targets the specific effect property path. The path construction follows AE's property hierarchy:

```
ADBE Effect Parade / ADBE Gaussian Blur 2 / ADBE Gaussian Blur 2-0001
```

The bridge's `setPropertyValue` command navigates this hierarchy using the `_aeHubGBP` (Get By Path) helper. Ensure that the property path sent from the UI matches the exact path structure the bridge expects. The path segments are typically the match names of each group/property level.

**Critical:** When setting a value within an effect, the undo group should wrap the entire operation. Single value changes within effects should be individually undoable (each change creates one undo step in AE). Don't wrap multiple effect property changes in a single undo group unless they're part of a deliberate batch operation (like applying a preset).

### 36.2 — Shape Layer Content Tree

A shape layer's content is a hierarchical tree:

```
Shape Layer
├── Contents
│   ├── Group 1
│   │   ├── Rectangle Path 1
│   │   ├── Fill 1
│   │   ├── Stroke 1

│       ├── Fill 3
```

Each node in this tree has a match name that identifies its type. The UI must render this tree with **type-specific labels and controls**, not generic property names.

### 36.5 — Shape Tree Rendering — Implementation Approach

The shape tree renderer must be an **enhanced version** of the existing recursive property renderer. The enhancement adds a type-detection step before rendering each node:

```
function renderShapeNode(node, container) {
  const label = shapeOperatorLabels[node.matchName] || node.name;
  const type = identifyShapeOperatorType(node.matchName);
  
  switch (type) {
    case 'rectangle':
      renderRectangleControls(node, container, label);
      break;
    case 'ellipse':
      renderEllipseControls(node, container, label);
      break;
    case 'fill':
      renderFillControls(node, container, label);
      break;
    case 'stroke':
      renderStrokeControls(node, container, label);
      break;
    case 'trim':
      renderTrimPathsControls(node, container, label);
      break;
    default:
      // Fall back to generic recursive renderer
      renderGenericPropertyGroup(node, container, label);
      break;
  }
  
  // Render children recursively (for groups, etc.)
  if (node.children) {
    node.children.forEach(child => renderShapeNode(child, childContainer));
  }
}
```

This approach gives specialized rendering for known operators while falling back gracefully for unknown ones.

## 37. CAMERA PANEL — COMPLETE SPECIFICATION

### 37.1 — Camera Property Categories

AE cameras have properties organized into two groups: camera options and transform.

**Camera Options:**
- **Zoom:** Scrub dial. Controls the camera's zoom level. Units: pixels. Range: depends on comp resolution, typically 100-10000+.
- **Depth of Field:** Toggle switch. Enables/disables DOF.
- **Focus Distance:** Scrub dial. Only interactive when DOF is enabled. Range: 0 to comp diagonal.
- **Aperture:** Scrub dial. Controls the iris size. Only interactive when DOF is enabled. Units: pixels.
- **Blur Level:** Scrub dial. Controls the DOF blur intensity. Only interactive when DOF is enabled. Range: 0-100%.
- **Film Size:** Scrub dial. The film gate size. Units: pixels.

**Transform:**
- **Position (X, Y, Z):** Three scrub dials (standard 3D transform).
- **Point of Interest (X, Y, Z):** Three scrub dials. Only relevant when the camera is set to "One-Node Camera" vs "Two-Node Camera". For two-node cameras, the point of interest controls where the camera looks.
- **Orientation (X, Y, Z):** Three scrub dials. Euler angles for 3D orientation.
- **X Rotation, Y Rotation, Z Rotation:** Individual axis rotations (separate from orientation).

### 37.2 — Conditional Control Activation

When Depth of Field is OFF:
- Focus Distance, Aperture, and Blur Level controls should be **visually disabled** (grayed out, non-interactive). This prevents the user from adjusting values that have no visible effect and reduces confusion.

When Depth of Field is ON:
- All DOF controls become active and interactive.

This conditional activation should be driven by the DOF toggle value. When the toggle is changed, immediately update the enabled/disabled state of dependent controls:

```
function updateDOFControls(dofEnabled) {
  focusDistanceControls.forEach(el => el.disabled = !dofEnabled);
  apertureControls.forEach(el => el.disabled = !dofEnabled);
  blurLevelControls.forEach(el => el.disabled = !dofEnabled);
  // Visual: reduce opacity on disabled controls
}
```

### 37.3 — Two-Node vs One-Node Camera

AE cameras can be "one-node" (no point of interest, just position and orientation) or "two-node" (position + point of interest). The bridge may or may not return information about which type the camera is. If the bridge returns the camera type, conditionally show/hide the Point of Interest controls. If the bridge doesn't return this info, show POI controls always — they simply won't have any visual effect on a one-node camera, which is a non-destructive state.

### 37.4 — Camera Presets (Future Enhancement)

Common camera presets: "Default 50mm", "Wide 24mm", "Telephoto 135mm", "Extreme Close-Up". Each preset would set zoom, aperture, and potentially DOF settings to match the real-world lens equivalent. This is a nice-to-have that adds value for cinematographers using Touch Tools.

### 38.3 — Source Text Area

A `<textarea>` element that displays the current source text content of the selected text layer. The user can edit the text directly. This is the **primary interaction point** of the text panel — everything else is secondary.

**Behavior:**
- On layer selection (or tab switch to Text tab with a text layer selected): read the source text from the layer properties and populate the textarea.
- On **input** event (not on every keystroke — use a debounce of 500ms): send the new text content to AE via `setPropertyValue` targeting the source text property. The debounce prevents sending a command on every keystroke, which would be expensive for long text.
- On **blur** (leaving the textarea): send the final text content immediately (cancel any pending debounce timer). This ensures the last edit is committed.
- On **Enter** (for single-line text): AE text layers can be single-line. The textarea should detect whether the text layer is single-line or multi-line (from the layer's text properties) and adjust behavior accordingly. For single-line, Enter commits the edit and blurs the textarea. For multi-line, Enter adds a new line (standard textarea behavior).

**Font formatting limitation:** AE's source text supports mixed formatting (different fonts, sizes, colors within the same text layer). The textarea is plain text — it cannot represent mixed formatting. For basic text editing, this is acceptable. The user edits the raw text content, and formatting is applied separately through the paragraph/character style properties. For advanced mixed-formatting editing, a rich text editor would be needed — but that's a massive undertaking and out of scope.

**Text color:** A color swatch + picker, identical to the fill color control in the shapes panel. Maps to the text layer's fill color.

**Leading (line spacing):** Scrub dial or number input. Controls the spacing between lines. Range: 0 to 1000 (or Auto).

**Baseline Shift:** Scrub dial or number input. Shifts text vertically relative to the baseline. Range: -500 to 500.

### 40.4 — Property Dropdown

**This is the most critical improvement over the current implementation.** Instead of a text input for manual property path entry, provide a dropdown that is populated from the layer's property tree.

   ```
   "Effects > Gaussian Blur > Blurriness"
   "Contents > Group 1 > Fill 1 > Color"
   ```
4. Populate the dropdown with this list.

- **Value:** The keyframe value. For number properties: a single number. For array properties: comma-separated values (e.g., "960, 540"). For color: hex code.

### 43.4 — Implementation Approach

Attach a `keydown` listener on `document`. Check if the active element is an input/textarea (in which case, most shortcuts should NOT fire — typing "j" in a text field shouldn't trigger "previous keyframe"). Only fire shortcuts when the active element is the body or a non-input element:

```
document.addEventListener('keydown', (e) => {
  // Skip if user is typing in an input
  if (['INPUT', 'TEXTAREA', 'SELECT'].includes(e.target.tagName)) return;
  
  // Skip if modifier keys suggest a browser shortcut
  if (e.metaKey && e.key !== 'z') return; // allow Cmd+Z through
  
  switch (e.key) {
    case ' ':
      e.preventDefault();
      playPause();
      break;
    case 'j':
      goToPreviousKeyframe();
      break;
    // ... etc.
  }
});
```

### 43.5 — Shortcut Conflict Avoidance

Some shortcuts (like Space for play/pause) conflict with browser behavior (Space scrolls the page). Always call `e.preventDefault()` for shortcuts that Touch Tools handles. This prevents the browser from performing its default action.

For `Cmd/Ctrl + S` (save), prevent the browser's "Save Page As" dialog and instead send `saveProject` to AE. This is a common pattern in web apps.

### 43.6 — Shortcut Reference

The Help button (section 2.8) should include a keyboard shortcut reference in its guided tour or help modal. A simple table listing all shortcuts with their actions. This helps discoverability — users won't know about shortcuts unless they're documented somewhere in the UI.

## 45. ERROR HANDLING — COMPREHENSIVE STRATEGY

### 45.1 — Command Timeout Errors

When a command times out (12 seconds), the existing system rejects the promise. The `aeCommand` wrapper catches this and shows a toast: "Command timed out: [command name]". The user knows something went wrong.

**Improvement:** After a timeout, check the connection status. If the status has dropped to amber or red, the toast should say "Connection lost during command" instead of just "timed out". This gives the user actionable information (reconnect) rather than a vague timeout message.

### 45.2 — AE Script Errors

When an ExtendScript command fails (e.g., trying to set an invalid property value), AE returns an error message through the bridge. The `aeCommand` wrapper catches this and shows the AE error message in a toast.

**Improvement:** For common errors, provide user-friendly translations:
- "Unable to set value" → "Cannot set this property. The layer may be locked or the property may be expression-driven."
- "Invalid object" → "The selected layer or composition no longer exists. Try refreshing the layer list."
- "Out of range" → "The value is outside the allowed range for this property."

Build a mapping of common AE error strings to user-friendly messages. Fall back to the raw error for unmapped strings.

### 45.3 — Connection Errors

When the WebSocket connection fails:
- On initial connect: show error toast and remain in disconnected state
- During operation: show warning toast "Connection lost" and begin auto-reconnect
- On reconnect failure: silent (no toast for each failed attempt)
- On reconnect success: show success toast "Reconnected"

### 45.4 — Graceful Degradation

If a feature requires a bridge command that doesn't exist (e.g., `renameComp`), the command will timeout after 12 seconds. Rather than letting every unsupported command timeout:

- Build a **feature detection mechanism**: on initial connection, send a `getSupportedCommands` command (or `getAppInfo` which returns bridge capabilities). Store the result. When the user triggers a feature that requires an unsupported command, show an immediate toast: "This feature requires a bridge update" instead of waiting for a timeout.

- If `getSupportedCommands` doesn't exist in the bridge, fall back to the timeout behavior. But document the command as a recommended bridge enhancement.

## 48. UI POLISH — SPACING AUDIT

### 48.1 — Spacing System Enforcement

Every spacing value in the codebase must derive from the CSS custom property scale defined in section 6.1. The scale is: xs=4px, sm=8px, md=12px, lg=16px, xl=24px. No magic numbers. No arbitrary pixel values that don't belong to the scale. If a specific layout requires a value between two scale steps (e.g., 10px), use the nearest scale step (either sm=8 or md=12) rather than introducing a one-off value. Consistency in spacing is the single most impactful UI polish factor — inconsistent margins and padding create visual noise even when the user can't articulate what feels "off."

**Audit procedure:** Grep the CSS file for all pixel values. Replace every value that doesn't match the scale with the nearest scale value. Exceptions: border widths (1px, 2px), icon sizes (16px, 20px, 24px), and input heights (36px desktop, 44px touch). These are functional dimensions, not spacing.

### 48.2 — Container Padding

Every container (panel, card, header bar, expanded section) must have consistent internal padding. The standard padding is `var(--space-md)` (12px) on all sides. Tighter containers (scrub dial rows, button groups) use `var(--space-sm)` (8px). Generous containers (the main panel content area) use `var(--space-lg)` (16px).

**Layer row internal padding:** `0 var(--space-sm)` (8px horizontal, 0 vertical) — rows are fixed height so vertical padding is handled by the row height and vertical centering via flexbox alignment.

**Effect card internal padding:** `var(--space-sm)` on the header row, `var(--space-md)` on the expanded body. The slight increase in body padding creates visual breathing room between the card header and the property controls.

### 48.3 — Element Gap Consistency

All flex and grid containers must use `gap` (not margins) for inter-element spacing. The gap values follow the same scale:

- Header bar elements: `gap: var(--space-sm)` (8px)
- Scrub dial rows within a group: `gap: var(--space-xs)` (4px) — tight because dials are vertically stacked
- Button groups (transport, creation): `gap: 0` (no gap — buttons share borders or touch)
- Panel sections: `gap: var(--space-lg)` (16px) — generous separation between major sections
- Form input rows (label + input): `gap: var(--space-sm)` (8px)
- Tab strip buttons: `gap: 0` (tabs share edges for a unified strip appearance)

### 48.4 — Border Hierarchy

Borders serve as visual separators in the dark theme. Use exactly three border styles:

1. **Primary separator:** `1px solid var(--bg-elevated)` — between major sections (panel header from content, tab strip from panel)
2. **Subtle separator:** `1px solid rgba(255,255,255,0.04)` — between minor elements (layer rows, scrub dial rows)
3. **Accent border:** `2px solid var(--accent)` — for selected/active states only (selected layer row left edge, active tab bottom edge)

No other border styles. No `border: 1px solid #333` with hardcoded values. No double borders. No inset/outset borders.

### 48.5 — Border Radius Consistency

Use exactly two radius values:
- `var(--radius-sm)` (4px): Inputs, buttons, small elements
- `var(--radius-md)` (6px): Cards, dropdowns, panels, context menus

No rounded pills (large radius on buttons). No sharp corners (0px radius). The consistent rounding creates a cohesive visual language.

## 49. UI POLISH — TYPOGRAPHY

### 49.1 — Font Stack Definition

Three font roles, each with a specific purpose:

**Display font (headings, brand):** Used for "Touch Tools" brand text, section headers, and panel titles. Choose a font with personality — something like Syne, Instrument Serif, or DM Serif Display. Loaded via Google Fonts CDN with `<link rel="preconnect">` and `<link href>` in the HTML `<head>`.

**Body font (UI text, labels, descriptions):** Used for all standard UI text — button labels, layer names, property labels, dropdown options, toast messages. Choose a clean, legible font — Lora, Source Serif 4, or Crimson Pro for serif; DM Mono's proportional sibling or a clean sans for sans-serif. Must be highly legible at small sizes (11-13px) on both desktop and RDP.

**Mono font (values, timecodes, code):** Used for numeric values in scrub dials, timecode display, frame counter, script console output, property paths. Must be monospaced with clear distinction between 0/O, 1/l/I. DM Mono or JetBrains Mono are strong choices.

### 49.2 — Type Scale

Define exactly six size steps:

| Step | Size | Usage |
|------|------|-------|
| xs | 10px | Layer index, flag labels, min/max labels on scrub dials |
| sm | 11px | Secondary labels, muted text, dropdown sub-text |
| base | 12-13px | Primary UI text, button labels, input text, layer names |
| md | 14px | Section headers, timecode display |

## 50. UI POLISH — COLOR CONSISTENCY

### 50.1 — Surface Hierarchy

Exactly four surface levels, each darker or lighter than the previous:

1. **Body background** (`--bg-body: #0a0a0a`): The outermost background. Visible only in the margins of the layout (which should be minimal since the app fills the viewport).
2. **Surface** (`--bg-surface: #141414`): The main panel background. Used for the content area of each tab panel.
3. **Elevated** (`--bg-elevated: #1e1e1e`): Cards, dropdowns, context menus, expanded layer sections. Slightly lighter than surface to appear "above" it.
4. **Input** (`--bg-input: #1a1a1a`): Form inputs, textareas, dropdowns. Between surface and elevated in brightness.

### 50.2 — Accent Color Usage Rules

The accent color has exactly five permitted uses:
1. **Selected state indicator:** Left border on selected layer row, bottom border on active tab
2. **Active toggle/button state:** Background on active flag toggles, active trackpad mode button
3. **Scrub dial fill:** The filled portion of the track from minimum to handle position
4. **Keyframe diamond:** Filled diamond when keyframe exists at current time
5. **Links and interactive hints:** The pencil icon on editable names, the help button

The accent color is NEVER used for:
- Body text or general labels
- Borders on non-selected elements
- Backgrounds of non-interactive containers
- Scrollbar thumbs
- Toast notifications (those use semantic colors)

### 50.3 — Semantic Color Usage

- **Success (#00e676):** Connection status dot (green), success toasts
- **Warning (#ffc107):** Connection status dot (amber), warning toasts, reconnecting messages
- **Error (#ff1744):** Connection status dot (red), error toasts, delete button hover state, validation error borders
- **Info:** No dedicated color — uses the default text color with a subtle surface background tint

### 50.4 — Hover and Active States

Every interactive element must have visible hover and active states. The pattern:

- **Hover:** Lighten the background by one step (e.g., surface → elevated, elevated → slightly brighter). Use `transition: background var(--transition-fast)` for smooth change.
- **Active (pressed):** Darken slightly from the hover state or apply a subtle `transform: scale(0.98)` to simulate physical button depression.
- **Focus (keyboard):** An `outline: 2px solid var(--accent); outline-offset: 2px;` visible only when navigating via keyboard (`:focus-visible` pseudo-class). Never show focus rings for mouse clicks — they're visually distracting for non-keyboard users.

### 51.1 — Button Click Feedback

This creates a tactile "click" feel. Without it, buttons feel unresponsive, especially over RDP where there's inherent latency.

### 51.3 — Expand/Collapse Transitions

When a section expands (effect card body, layer expand, Tier 2 transform group, comp settings panel), use `max-height` transition:

```
.section-body {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.25s ease;
}

.section-body.expanded {
  max-height: 800px; /* generous max, actual content determines real height */
}
```

**Important caveat:** `max-height` transitions have a known issue — if the actual content height is much less than the max-height value, the animation appears to complete instantly (because the transition from 0 to 800px animates over 250ms, but if content is only 200px tall, it reaches its final height quickly and the remaining animation is invisible). To mitigate: set the max-height as close to the actual content height as possible. This requires measuring the content height with JavaScript (`element.scrollHeight`) before applying the expanded class:

```
function toggleSection(bodyElement) {
  if (bodyElement.classList.contains('expanded')) {
    bodyElement.style.maxHeight = bodyElement.scrollHeight + 'px';
    requestAnimationFrame(() => {
      bodyElement.style.maxHeight = '0';
    });
    bodyElement.classList.remove('expanded');
  } else {
    bodyElement.classList.add('expanded');
    bodyElement.style.maxHeight = bodyElement.scrollHeight + 'px';
  }
}
```

After the transition ends, set `max-height` to `none` (for the expanded state) so the content can grow dynamically without being clipped. Listen for `transitionend` event to apply this.

## 52. PERFORMANCE OPTIMIZATION AUDIT

### 52.1 — DOM Operation Batching

Every function that creates or modifies multiple DOM elements must batch its operations. The single most expensive DOM operation is layout recalculation (reflow), which happens when the browser needs to compute the position and size of elements after a DOM change.

**Pattern:** Build the complete DOM tree in a `DocumentFragment`, then append it to the live DOM in a single operation:

```
function renderLayerList(layers) {
  const fragment = document.createDocumentFragment();
  layers.forEach((layer, i) => {
    const row = createLayerRow(layer, i);
    fragment.appendChild(row);
  });
  layerListContainer.innerHTML = '';
  layerListContainer.appendChild(fragment);
  // Single reflow instead of N reflows
}
```

This is especially critical for the layer list (which could have 50+ rows) and the property tree (which could have 100+ scrub dials).

### 52.2 — Selective DOM Updates

As noted in section 6.3, after setting a single property value, don't re-render the entire property tree. Instead, locate the specific DOM element for that property and update only its value display and handle position:

```
function updateSinglePropertyValue(propertyPath, newValue) {
  const valueInput = document.querySelector(`[data-property-path="${propertyPath}"] .value-input`);
  const handle = document.querySelector(`[data-property-path="${propertyPath}"] .scrub-handle`);
  if (valueInput) valueInput.value = newValue;
  if (handle) {
    // Recalculate handle position from new value
    const fraction = (newValue - min) / (max - min);
    handle.style.left = (fraction * trackWidth) + 'px';
  }
}
```

This is an O(1) DOM operation instead of O(N) where N is the total number of properties.

### 52.3 — Event Listener Count

After implementing event delegation (section 8), the total event listener count should be:

| Target | Listeners | Purpose |
|--------|-----------|---------|
| `document` | 1 keydown | Global keyboard shortcuts |
| `document` | 1 click | Context menu dismiss |
| Header bar | 1 click | Header button delegation |
| Tab strip | 1 click | Tab switching delegation |
| Layer list container | 1 click | Layer selection, flag toggles, action delegation |
| Layer list container | 1 contextmenu | Right-click context menu |
| Layer list container | 2 touchstart/touchend | Long-press detection |
| Property panel container | 1 pointerdown | Scrub dial delegation |
| Trackpad element | 1 pointerdown | Trackpad drag initiation |
| Trackpad element | 1 wheel | Scroll wheel support |
| Transport container | 1 click | Transport button delegation |
| Search input | 1 input | Effect search with debounce |
| Script console input | 1 keydown | Command execution on Enter |

**Total: ~15 event listeners.** Compare this to the non-delegated approach which could easily reach 200+ listeners on a complex layer with many properties. The delegation approach is roughly 13x more efficient in event listener count.

### 52.4 — CSS Selector Efficiency

CSS selectors are parsed right-to-left. Complex selectors force the browser to do more work during style recalculation. Guidelines:

- **Use class selectors** exclusively. No tag selectors (`.layer-row` not `div.row`), no attribute selectors (`[data-tab]` not `[data-tab="transform"]`), no descendant combinators spanning multiple levels (`.panel .content .row` — too deep).
- **Avoid universal selectors** (`*`) in performance-sensitive contexts. The global reset `* { box-sizing: border-box; }` is acceptable because it runs once.
- **Avoid pseudo-selectors on dynamic elements.** Don't use `.layer-row:nth-child(2n):hover:active` — too many conditions to evaluate per frame during interactions.
- **Use `will-change: transform`** on elements that animate frequently (scrub dial handles, toggle switch thumbs, playhead scrubber handle). This tells the browser to promote these elements to their own compositor layer, avoiding main-thread layout work during animation.

### 52.5 — Memory Management

**Detach event listeners on element removal.** When the layer list is re-rendered (new comp selected, refresh button clicked), the old layer rows are removed from the DOM. Event delegation on the container handles this automatically (the container isn't removed, only its children). But any custom data stored on elements (dataset attributes, attached objects) should be cleaned up.

**Don't store large data in DOM attributes.** The property tree from AE can be large. Store it in a JavaScript variable (e.g., `currentPropertyTree`) and reference it by key from DOM elements, rather than serializing the entire tree into data attributes on each DOM node.

**Clear intervals and timeouts on disconnect.** The transport polling interval (500ms `setInterval` for `getCurrentTime`) must be cleared when the connection drops. Otherwise it fires indefinitely, attempting to send commands that will fail. The existing code may handle this, but verify:

```
let transportInterval = null;

function startTransportPolling() {
  stopTransportPolling(); // clear any existing
  transportInterval = setInterval(refreshTransport, 500);
}

function stopTransportPolling() {
  if (transportInterval) {
    clearInterval(transportInterval);
    transportInterval = null;
  }
}
```

Call `stopTransportPolling()` in the disconnect handler.

### 52.6 — Command Coalescing

When multiple value changes happen rapidly (scrub dial drag, trackpad movement), RAF throttling limits sends to ~60fps. But some commands may still queue up if AE is slow to respond. Implement a **pending command deduplication**:

If a `setPropertyValue` command for the same property path is already pending (sent but not yet resolved), and a new value arrives for the same property, update the pending command's value rather than queuing a new command. The bridge will process the latest value, and earlier intermediate values are skipped.

This is more aggressive than simple throttling — it ensures that for any given property, at most ONE command is in-flight at any time. The resolution of that command triggers the next send if the value has changed again.

```
const pendingPropertyCommands = new Map(); // path -> {id, value}

function setPropertyValueThrottled(path, value, layerIndex) {
  if (pendingPropertyCommands.has(path)) {
    // Update the pending value — the command is already in-flight
    // but we'll queue the latest value for when it resolves
    pendingPropertyCommands.get(path).nextValue = value;
    return;
  }
  
  const promise = send({command: 'setPropertyValue', args: {path, value, layerIndex}});
  const entry = {id: promise.id, nextValue: null};
  pendingPropertyCommands.set(path, entry);
  
  promise.then(() => {
    pendingPropertyCommands.delete(path);
    if (entry.nextValue !== null) {
      // A newer value arrived while we were waiting — send it now
      setPropertyValueThrottled(path, entry.nextValue, layerIndex);
    }
  }).catch(() => {
    pendingPropertyCommands.delete(path);
  });
}
```

## 53. TOUCH COMPATIBILITY

### 53.1 — Viewport Meta Tag

The HTML must include the viewport meta tag to prevent mobile browsers from zooming and to set proper scaling:

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```

`user-scalable=no` prevents pinch-to-zoom on the entire page, which would interfere with the trackpad's pinch-to-scale gesture and scrub dial manipulation.

### 53.2 — Touch Target Sizing

Every interactive element must meet the minimum touch target size. The WCAG recommendation is 44×44 CSS pixels. This applies to:

- All buttons (creation buttons, transport buttons, flag toggles, context menu items, tab buttons)
- All inputs (text inputs, number inputs, search input)
- All toggles (switch components for booleans)
- All dropdown triggers (comp selector, blend mode, mask mode)
- Scrub dial hit areas (the track should be at least 44px tall as a hit area, even if the visual track is only 3-4px tall)

**Implementation:** For visual elements smaller than 44px (like a 12px scrub handle), add invisible padding to increase the hit area:

```
.scrub-hit-area {
  position: absolute;
  top: -16px;
  bottom: -16px;
  left: 0;
  right: 0;
  /* 44px total height for a 12px visual handle */
}
```

### 53.3 — Touch Event Handling

Touch interactions differ from mouse interactions in several ways:

**No hover state:** Touch devices don't have hover. Any functionality that relies on hover (showing the pencil icon on editable names, showing tooltips, expanding on hover) must have an alternative for touch. The pencil icon should always be visible on touch devices. Tooltips should be triggered by tap (with auto-dismiss). Expand should be tap-only, never hover.

**Detection:** Use `@media (pointer: coarse)` in CSS for touch-specific styles. In JavaScript, check `'ontouchstart' in window` or use the pointer events API which abstracts over mouse and touch.

**Long-press for context menu:** As specified in section 42.6, touch devices use long-press (500ms hold) instead of right-click.

**Scroll vs. drag disambiguation:** On touch, a vertical finger movement could be a scroll (of the panel) or a drag (of a scrub dial or trackpad). The disambiguation is based on which element the touch starts on: touching a scrub dial handle initiates a drag; touching the panel background initiates a scroll. The scrub dial's hit area should be clearly defined so touch events are captured correctly.

**Prevent pull-to-refresh:** On mobile Chrome and Safari, pulling down from the top of the page triggers a page refresh. Since the app should never refresh during normal operation, prevent this with `overscroll-behavior: none` on the body:

```css
body {
  overscroll-behavior: none;
}
```

### 53.4 — Virtual Keyboard Handling

When the user taps an input field (layer name, search, value input, script console), the virtual keyboard appears and reduces the visible viewport. The layout must handle this gracefully:

- The header bar and tab strip should remain visible (sticky positioning handles this)
- The active panel should scroll to ensure the focused input is visible
- Use the `visualViewport` API to detect keyboard appearance and adjust layout if needed

```
window.visualViewport.addEventListener('resize', () => {
  const keyboardHeight = window.innerHeight - window.visualViewport.height;
  if (keyboardHeight > 100) {
    // Keyboard is visible — adjust layout
    document.body.style.height = window.visualViewport.height + 'px';
  } else {
    document.body.style.height = '100vh';
  }
});
```

### 53.5 — Orientation Handling

On tablets, the user may switch between portrait and landscape orientation. The responsive breakpoints (section 11) handle layout changes, but also consider:

- In landscape: maximize horizontal space — wider scrub dials, wider trackpad
- In portrait: maximize vertical space — more compact rows, smaller spacing, collapsible sections defaulting to collapsed

Listen for `orientationchange` or `resize` events and adjust the layout accordingly. The CSS media queries handle most of this automatically, but the trackpad size should recalculate its crosshair position and sensitivity on resize.

## 54. ACCESSIBILITY

### 54.1 — ARIA Attributes

All interactive elements must have appropriate ARIA attributes:

- **Buttons:** `role="button"` (implicit for `<button>`, explicit for custom elements), `aria-label` for icon-only buttons (e.g., `<button aria-label="Undo">↩</button>`)
- **Toggles:** `role="switch"`, `aria-checked="true/false"` to communicate the toggle state to screen readers
- **Tabs:** `role="tablist"` on the container, `role="tab"` on each button, `role="tabpanel"` on each content area, `aria-selected="true/false"` on the active tab
- **Dropdown menus:** `role="menu"` on the container, `role="menuitem"` on each item, `aria-expanded="true/false"` on the trigger
- **Layer list:** `role="listbox"` on the container, `role="option"` on each row, `aria-selected="true/false"` for the selected layer
- **Context menu:** `role="menu"`, `aria-expanded`, and focus management (focus should move into the menu when it opens, and return to the trigger element when it closes)
- **Live regions:** Connection status changes, toast notifications, and error messages should be announced to screen readers using `aria-live="polite"` on the relevant container

### 54.2 — Focus Management

**Tab order:** The natural tab order should follow the visual layout: header bar → comp settings (when open) → comp selector → timeline → tab strip → active panel content. This means elements in the HTML source order should match the visual order.

**Focus trapping for modals:** When a modal (context menu, dropdown, confirmation dialog) is open, Tab key should cycle within the modal only, not escape to elements behind it. Implement a focus trap:

```
function trapFocus(modalElement) {
  const focusable = modalElement.querySelectorAll('button, input, select, [tabindex]');
  const first = focusable[0];
  const last = focusable[focusable.length - 1];
  
  modalElement.addEventListener('keydown', (e) => {
    if (e.key === 'Tab') {
      if (e.shiftKey && document.activeElement === first) {
        e.preventDefault();
        last.focus();
      } else if (!e.shiftKey && document.activeElement === last) {
        e.preventDefault();
        first.focus();
      }
    }
  });
}
```

**Focus restoration:** When a modal or context menu closes, focus should return to the element that triggered it. Store the trigger element reference before opening the modal, and call `.focus()` on it after closing.

### 54.3 — Color Contrast

All text must meet WCAG AA contrast requirements against its background:

- `--text-primary` (#e8e4dc) on `--bg-surface` (#141414): contrast ratio ~12:1 — exceeds AAA ✓
- `--text-secondary` (#9a9590) on `--bg-surface` (#141414): contrast ratio ~5:1 — meets AA ✓
- `--text-muted` (#5a5550) on `--bg-surface` (#141414): contrast ratio ~2.5:1 — **fails AA** ✗

**Fix:** The muted text color must be lightened to at least #706b65 to meet the 4.5:1 AA ratio for normal text. If the muted color is purely decorative (not essential information), the AA failure is acceptable under WCAG's "incidental" exception. But for anything the user needs to read (min/max labels, disabled text), ensure compliance.

### 54.4 — Reduced Motion

Some users prefer reduced motion (vestibular disorders, motion sensitivity). Respect the `prefers-reduced-motion` media query:

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

This effectively disables all animations and transitions for users who have enabled the reduced motion preference in their OS settings. The functionality is preserved — only the motion is removed.

### 54.5 — Screen Reader Announcements

Key state changes should be announced via `aria-live` regions:

- Connection state change: "Connected to After Effects" / "Disconnected from After Effects"
- Layer selection: "Selected layer: [layer name]"
- Command errors: "Error: [error message]"
- Command success (for destructive operations): "Layer deleted" / "Effect removed"

Create a visually hidden `aria-live="polite"` div that receives these announcements:

```html
<div aria-live="polite" class="sr-only" id="announcer"></div>
```

```css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
```

## 55. EDGE CASES — COMPREHENSIVE CATALOG

### 55.1 — Empty States

Every panel must have a defined empty state. When there is no data to display, show a helpful message instead of a blank space:

| Panel | Empty State Message |
|-------|-------------------|
| Layer list | "No layers in this composition. Create a layer using the buttons above." |
| Effects | "No effects on this layer. Search for an effect above to get started." |
| Shapes | "Select a shape layer to view its contents." |
| Camera | "Select a camera layer to view its properties." |
| Text | "Select a text layer to edit its content." |
| Masks | "No masks on this layer. Click 'Add Mask' to create one." |
| Keyframes | "Select a layer and property to view keyframes." |
| Script console output | "(no output)" in muted italic |

### 55.2 — No Layer Selected

When no layer is selected (deselected all, or just connected with no user interaction yet):
- Transform panel shows: "Select a layer to view its properties."
- Effects panel shows: "Select a layer to view its effects."
- All other panels show their empty state messages.
- Trackpad is disabled (visual dimming, no interaction response).
- Undo/Redo buttons remain functional (they operate at the application level, not layer level).

### 55.3 — Multiple Comps With Same Name

AE allows multiple compositions to have the same name. The comp dropdown must handle this. If two comps have the same name, append the comp's index or a distinguishing identifier: "Main Comp (1)", "Main Comp (2)". Or show the comp's ID from the bridge response as a suffix: "Main Comp #3".

### 55.4 — Very Long Layer Names

Layer names can be very long. The layer name display must handle this gracefully:
- Truncate with ellipsis: `text-overflow: ellipsis; white-space: nowrap; overflow: hidden;`
- The full name is visible in the editable input when the user clicks to rename
- Consider adding a `title` attribute with the full name for native tooltip on hover (desktop)

### 55.5 — Deep Property Trees

Some layers (especially those with many effects or complex shape structures) can produce very deep property trees. The recursive renderer must handle arbitrary depth without:
- Stack overflow (JavaScript recursion limit is typically 10,000+ levels, but AE property trees rarely exceed 10 levels — this is theoretical, not practical)
- Performance degradation (rendering 500+ scrub dials simultaneously)
- Layout breaking (extreme indentation from deeply nested groups)

**Mitigation:** Cap the visible nesting depth at 6 levels. Beyond that, show a "More properties..." link that opens a sub-panel or scrolls to the deeply nested content. For most practical AE compositions, 6 levels is sufficient.

### 55.6 — Rapid Tab Switching

If the user clicks tabs rapidly (Effects → Shapes → Camera → Text within a second), multiple `load*()` commands fire in quick succession. The responses may arrive out of order, causing the wrong panel to display data intended for a different panel.

**Mitigation:** Each `load*()` function should check if its tab is still the active tab when the response arrives. If the user has switched away, discard the response:

```
async function loadEffects() {
  const expectedTab = 'effects'; // the tab this function serves
  const response = await aeCommand('getEffects', {...});
  if (state.activeTab !== expectedTab) return; // user switched away — discard
  renderEffects(response);
}
```

### 55.7 — Bridge Disconnection During Command

If the TCP bridge disconnects while a command is in-flight, the command will never resolve. The existing 12-second timeout handles this — the command promise rejects after 12 seconds. But the user experience can be improved:

When the connection status transitions from green to amber/red, immediately reject all pending commands with a specific error: "Connection lost". This gives instant feedback rather than making the user wait 12 seconds for a timeout.

```
function onConnectionLost() {
  pending.forEach((entry, id) => {
    entry.reject(new Error('Connection lost'));
    clearTimeout(entry.timer);
  });
  pending.clear();
}
```

### 55.8 — AE Application Not Responding

If AE is processing a heavy operation (rendering, complex expression evaluation), it may not respond to bridge commands promptly. The 12-second timeout will eventually fire, but the user may see the UI appear frozen.

**Mitigation:** Show a warning toast at 5 seconds: "AE is taking longer than expected to respond...". The command continues to wait until the 12-second timeout. If the command resolves between 5-12 seconds, dismiss the warning toast. If it times out, replace with the timeout error toast.

```
const SLOW_WARNING_MS = 5000;
const TIMEOUT_MS = 12000;

function sendWithTimeout(command) {
  return new Promise((resolve, reject) => {
    const warningTimer = setTimeout(() => {
      showToast('AE is taking longer than expected...', 5000);
    }, SLOW_WARNING_MS);
    
    const timeoutTimer = setTimeout(() => {
      reject(new Error('Command timed out'));
    }, TIMEOUT_MS);
    
    send(command).then(result => {
      clearTimeout(warningTimer);
      clearTimeout(timeoutTimer);
      resolve(result);
    }).catch(err => {
      clearTimeout(warningTimer);
      clearTimeout(timeoutTimer);
      reject(err);
    });
  });
}
```

### 55.9 — Browser Tab Visibility

When the browser tab is in the background, `setInterval` and `requestAnimationFrame` may throttle or pause. This affects:
- Transport polling (500ms interval)
- RAF-based throttle on scrub dials and trackpad

When the tab becomes visible again (via `document.visibilitychange` event), force an immediate transport refresh to sync the timecode:

```
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible') {
    if (state.connection === 'ae') {
      refreshTransport(); // immediate sync
      if (state.isPlaying) startTransportPolling(); // ensure interval is running
    }
  }
});
```

### 55.10 — WebSocket Reconnection State Consistency

When the WebSocket reconnects after a disconnection, the local state may be stale:
- The selected comp may no longer exist
- The selected layer may have been deleted or renamed
- The composition settings may have changed
- The playhead may have moved

**On reconnect:** Force a full state refresh:
1. `loadComps()` — refresh comp list
2. If the previously selected comp exists, `setComp()` to confirm it's still active
3. `loadLayers()` — refresh layer list
4. If a layer was selected, verify it still exists in the new layer list
5. `refreshTransport()` — sync timecode
6. Restart transport polling if playback was active

This ensures the UI is fully consistent with AE's current state after any disconnection.

## 59. FINAL NAMING CONVENTIONS

### 59.1 — Project Name

**Touch Tools** — all references updated in HTML title, any branding elements, README, package.json name field.

### 59.2 — CSS Class Naming

Use BEM-like naming (Block-Element-Modifier) without strict BEM syntax. The pattern: `block-element-modifier`. Examples:
- `.layer-row` — block
- `.layer-row--selected` — modifier (selected state)
- `.layer-name` — element within layer block
- `.scrub-dial-track` — block + element
- `.scrub-dial-handle--active` — modifier
- `.effect-card-header` — block + element
- `.context-menu-item--danger` — modifier

No camelCase in CSS classes. No single-letter classes (except utility classes). No IDs for styling (IDs only for JavaScript `getElementById` when needed for specific targeting).

### 59.3 — JavaScript Naming

- Functions: `camelCase` — `loadLayers`, `selectLayer`, `setPropertyValueThrottled`
- Constants: `UPPER_SNAKE_CASE` — `TIMEOUT_MS`, `SLOW_WARNING_MS`, `SENSITIVITY_FINE`
- State keys: `camelCase` — `selectedLayerIndex`, `currentCompId`, `isPlaying`
- DOM references: `camelCase` with descriptive names — `layerListContainer`, `timecodeDisplay`, `playheadHandle`
- Event handlers: `on` prefix — `onLayerClick`, `onTabSwitch`, `onConnectionLost`
- Async functions: verb-based — `loadComps`, `loadEffects`, `searchEffects` (no `fetch` prefix — that implies HTTP fetch)

### 59.4 — Data Attribute Naming

- `data-tab="transform"` — tab identification
- `data-layer-index="3"` — layer index
- `data-property-path="ADBE Transform Group/ADBE Position"` — AE property path
- `data-effect-index="0"` — effect index within layer
- `data-mask-index="1"` — mask index
- `data-action="split"` — action identification for event delegation
- `data-type="number"` — property type for specialized rendering

All lowercase, hyphen-separated.
